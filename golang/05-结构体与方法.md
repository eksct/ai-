# 结构体与方法

## 设计原理

### 为什么 Go 没有类

Go 故意不设计"类"——它认为面向对象的继承已经证明在大规模项目中弊大于利（复杂的继承层级难以理解、菱形继承问题、脆弱基类问题）。

Go 的方案：**用结构体 + 方法实现封装，用嵌入实现组合，用接口实现多态。**

```
Java 的继承：
  class Dog extends Animal implements Walkable { ... }
  → 继承了 Animal 的行为，实现了 Walkable 接口
  → Animal 改了内部实现，子类跟着变（脆弱基类问题）
  → Dog 可以赋值给 Animal 变量（多态），调用 speak() 看实际类型

Go 的嵌入：
  type Dog struct {
      Animal      // 嵌入
  }
  → Dog 拥有 Animal 的提升字段和方法（改了 Animal，Dog 同样受影响）
  → 但 Dog 不能赋值给 Animal 变量，不是"父子关系"
  → 没有方法覆盖（override），只有同名方法遮蔽（shadow），且编译期确定
```

### 值接收者 vs 指针接收者

这是 Go 新手最容易困惑的地方，选错有性能问题和 bug。

```go
// 值接收者：方法操作的是副本
func (u User) String() string {
    return u.Name  // 修改 u 不会影响原对象
}

// 指针接收者：方法操作的是原对象
func (u *User) SetName(name string) {
    u.Name = name  // 修改了原对象
}
```

**选型原则：**

```
需要修改接收者                → 指针（必须）
结构体很大（> 几十个字段）     → 指针（避免复制开销）
结构体包含 sync.Mutex 等      → 指针（Mutex 不能复制）
其他情况优先用值接收者         → 值接收者

坑：一个类型同时在值接收者和指针接收者混用 → 可以，但建议保持一致性
```

**经验之谈：** 大厂代码规范里有一条隐性规则——"拿不准的时候就用指针接收者"。原因：值接收者复制整个结构体，当时结构小没关系，几个月后有新人加了几个大字段，性能就崩了。指针接收者没有这个问题。

## 核心概念

### 嵌入 vs 继承

```go
type Animal struct {
    Name string
}

func (a Animal) Speak() string {
    return "..."
}

type Dog struct {
    Animal  // 嵌入（不是继承）
    Breed string
}

// 方法提升：Dog 可以直接调 Speak()
dog := Dog{Animal: Animal{Name: "旺财"}, Breed: "金毛"}
dog.Speak()  // "..."（从 Animal 提升上来的）

// 但不能多态赋值
// var a Animal = Dog{}  // 编译错误！Dog 不是 Animal
```

## 生产实践

### 结构体 Tag 的正确使用

```go
type ImageMetadata struct {
    ID        int64     `json:"id"       db:"id"        gorm:"primaryKey"`
    FileName  string    `json:"file_name" db:"file_name" gorm:"column:file_name;size:255"`
    Width     int       `json:"width"    db:"width"      validate:"min=1"`
    Height    int       `json:"height"   db:"height"     validate:"min=1"`
    Format    string    `json:"format"   db:"format"     validate:"required,oneof=jpg png gif webp"`
    FileSize  int64     `json:"file_size" db:"file_size" gorm:"-"`
    // gorm:"-" 表示忽略这个字段（比如计算字段）
}

// tag 用空格分隔，每种用途一个
// 格式：`key1:"val1" key2:"val2"`
// 不要用逗号——gorm 和 validate 的 options 用分号
```

**踩坑：** tag 拼写错误不会报编译错误（reflect 运行时才知道）。某次线上 `json:"width"` 写成了 `json:"widtht"`，API 响应里 width 字段丢了，前端白屏排查了半天。

### 嵌入的常见用途

```go
// 1. 数据模型加公共字段
type BaseModel struct {
    ID        int64     `json:"id"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}

type Image struct {
    BaseModel  // 嵌入公共字段
    FileName   string `json:"file_name"`
    FilePath   string `json:"file_path"`
}

// 2. 嵌入行为
type LogMixin struct {
    logger *log.Logger
}

func (l LogMixin) LogInfo(msg string) {
    l.logger.Printf("[INFO] %s", msg)
}

type Service struct {
    LogMixin  // 嵌入日志能力
}

svc := Service{LogMixin: LogMixin{logger: log.Default()}}
svc.LogInfo("service started")
```

### JSON 序列化的坑

```go
type Response struct {
    Code    int         `json:"code"`
    Data    interface{} `json:"data,omitempty"`
    Message string      `json:"message,omitempty"`
    // omitempty：零值时忽略这个字段
    // 但如果是 bool，false 会被忽略（可能不是你想要的）
}

// 时间格式化
type User struct {
    CreatedAt time.Time `json:"created_at"`
}
// 默认序列化为 RFC3339：2024-01-01T00:00:00Z
// 想要自定义格式：
type CustomTime time.Time
func (ct CustomTime) MarshalJSON() ([]byte, error) {
    return json.Marshal(time.Time(ct).Format("2006-01-02 15:04:05"))
}
```

### 结构体对齐优化

```go
// 内存布局
type ImageData struct {
    Width  int32   // 4 字节
    Height int32   // 4 字节
    Size   int64   // 8 字节
}
// 实际占用：4 + 4 + 8 = 16 字节（对齐良好）

// ❌ 不良布局
type BadImageData struct {
    Size   int64   // 8 字节
    Width  int32   // 4 字节
    Height int32   // 4 字节
}
// 多出 8 字节对齐填充

// 最佳实践：按类型大小降序排列
// int64 → int32 → int16 → bool
```
