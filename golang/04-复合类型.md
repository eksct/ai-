# 复合类型

## 设计原理

### 为什么 Go 的数组是值类型

C 语言数组是"退化为指针"，Java 数组是"引用类型"。Go 偏偏把数组设计成值类型——赋值就复制整个数组。

```
Java:
  int[] a = {1,2,3};
  int[] b = a;     // b 引用同一个数组对象
  b[0] = 100;       // a[0] 也变成 100

Go:
  a := [3]int{1,2,3}
  b := a            // b 复制了整个数组
  b[0] = 100        // a[0] 还是 1
```

**设计意图：** 避免"引用"导致的隐式共享。Go 偏好在显式传指针时才共享内存。这在并发场景下减少了"一个地方改了数组，另一个地方莫名其妙受影响"的问题。

**代价：** 大数组赋值性能差。这就是为什么 Go 几乎不用数组，全部用切片。

### 切片底层原理

```
切片 = 结构体 { ptr *Elem, len int, cap int }

s := []int{1, 2, 3}
ptr ──→ [1][2][3][ ]
len = 3
cap = 3

s = append(s, 4)
ptr ──→ [1][2][3][4]  // 容量满足，原地追加

s = append(s, 5, 6, 7, 8)  // 容量不够，分配新数组
ptr ──→ [1][2][3][4][5][6][7][8][ ][ ]
len = 8
cap = 16                // Go 扩容策略：小切片翻倍，大切片涨 1.25 倍
```

**踩坑记录：** 某次生产排查内存占用过高，发现是切片扩容导致底层数组频繁复制+未释放。处理大图片列表时，如果预知最终大小，一定要 `make([]Image, 0, estimated)` 提前分配容量。

### Map 的哈希冲突处理

Go 的 map 底层是哈希表，用链地址法解决冲突。和 Java 的 HashMap 类似，但一个关键区别：

```
Go 的 map 天生并发不安全：
  两个 goroutine 同时读写同一个 map → fatal error: concurrent map writes
  Java 的 ConcurrentHashMap 可以并发读写部分场景

为什么 Go 不提供并发安全的内置 map？
  设计者认为"加锁策略应该由业务决定"，内置实现了反而限制了选择。
```

实际生产中：
```go
// ✅ 读多写少 → sync.RWMutex
// ✅ 高频读写不同 key → sync.Map（做了读写分离优化）
// ✅ key 少，更新频繁 → 普通 map + sync.Mutex
// ✅ 不用的共享 → 不要共享（每个 goroutine 自己的 map）
```

## 核心概念

### 切片操作

```go
// 删除（Go 没有内置删除，需要自己拼）
func Remove[T any](s []T, i int) []T {
    return append(s[:i], s[i+1:]...)
}

// 插入
func Insert[T any](s []T, i int, v T) []T {
    s = append(s, zeroValue) // 扩容
    copy(s[i+1:], s[i:])     // 后移
    s[i] = v
    return s
}

// 过滤（不修改原 slice）
func Filter[T any](s []T, fn func(T) bool) []T {
    result := make([]T, 0, len(s))
    for _, v := range s {
        if fn(v) {
            result = append(result, v)
        }
    }
    return result
}
```

### 切片共享底层数组的陷阱

```go
// 场景：从大切片取一部分
allImages := readAllImages()  // 假设有 10000 张，占用 500MB
someImages := allImages[100:200]  // 取 100 张

// someImages 只引用了 100 张，但底层数组引用了 10000 张
// GC 不会释放那 9900 张的内存——有人引用着

// 修复：用 copy 解耦
someImages := make([]Image, 100)
copy(someImages, allImages[100:200])
// allImages 可以 GC 了
```

## 生产实践

### make 预分配的重要性

```go
// ❌ 不预分配
var users []User
for _, u := range all {
    users = append(users, u)  // 多次扩容，GC 压力大
}

// ✅ 预分配
users := make([]User, 0, len(all))
for _, u := range all {
    users = append(users, u)
}

// 性能差异：100 万条数据，不预分配耗时多 3-5 倍
```

### Map 遍历顺序

```go
// Go map 的遍历顺序是随机的——这是故意的
// 目的是防止开发者依赖遍历顺序

for k, v := range m {
    // 每次运行顺序可能不同
}

// 如果你需要稳定顺序：
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys)
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

### nil 切片 vs 空切片

```go
var s []int        // nil slice，len=0, cap=0
s := []int{}       // empty slice，len=0, cap=0

// 序列化结果不同
json.Marshal(nil)     // "null"
json.Marshal([]int{}) // "[]"

// API 响应中，"null" 和 "[]" 对前端的意义不同
// "null" 表示"没有这个字段"，"[]" 表示"数据为空"
// 建议统一用 make([]int, 0) 初始化，避免前端判断
```
