# 内存管理与性能优化

## 设计原理

### Go 的 GC 和 Java 的 GC 对比

```
Java GC：分代收集（Young/Old/Metaspace），STW 可长达几百 ms
Go GC：并发三色标记清除，目标 STW < 1ms
```

Go 选"低延迟"而非"高吞吐"——适合 API 网关、实时服务这种对延迟敏感的场景。Java 选"高吞吐"——适合批处理、离线计算这种不在乎偶尔卡顿的场景。

**代价：** Go 的 GC 在堆内存很大（> 100GB）时性能下降明显，吞吐不如 Java G1。

### 逃逸分析

```go
// Go 编译器决定变量分配位置，不是由 new 或 & 决定

// 栈上分配（不逃逸）
func createOnStack() User {
    return User{Name: "Alice"}  // 返回值，不是指针 → 栈分配
}

// 堆上分配（逃逸）
func createOnHeap() *User {
    return &User{Name: "Alice"}  // 返回指针，编译器不确定调用方行为 → 堆分配
}

// 查看逃逸分析
// go build -gcflags="-m" main.go
```

## 核心概念

### 内存布局

```go
// 结构体对齐规则
// Go 的字段对齐：按最大字段对齐

type Image struct {
    Width  int32  // 4 字节
    Height int32  // 4 字节
    // 4 + 4 = 8，对齐良好，不需要填充

    Size   int64  // 8 字节
}

type BadImage struct {
    Width  int32  // 4
    Size   int64  // 8 → 需要 4 字节填充
    Height int32  // 4 → 需要 4 字节对齐
    // 实际占用：4 + 4(pad) + 8 + 4 + 4(pad) = 24
    // 最优布局：4 + 4 + 8 = 16
}
```

### sync.Pool 原理

```go
// sync.Pool 不是缓存池，是"临时对象池"
// 它的对象可能在 GC 时被回收
// 适合存短期重复使用的大对象

var bufferPool = sync.Pool{
    New: func() any {
        return new(bytes.Buffer)
    },
}

func process(data []byte) {
    buf := bufferPool.Get().(*bytes.Buffer)
    defer bufferPool.Put(buf)
    buf.Reset()

    json.Compact(buf, data)  // 使用 buf
    // 用完后放回 pool，下次复用
}
```

## 生产实践

### Profiling 步骤

```bash
# 1. 在 main 中启动 pprof
import _ "net/http/pprof"

go func() {
    http.ListenAndServe("localhost:6060", nil)
}()

# 2. 采集性能数据（让程序跑一会再采集）
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# 3. 分析
(pprof) top10      # 前 10 热点
(pprof) web        # 火焰图（需要 graphviz）
(pprof) list func  # 看函数每行耗时
```

### 性能优化原则

```go
// 1. 先 profile，后优化（不要猜瓶颈在哪）
// 2. 业务代码的 95% 性能问题在：重复分配、不释放资源、不必要的内存复制
// 3. 编译器比你想象的聪明——写清晰代码，让编译器优化

// ❌ 过度优化
func (s *Service) Get() []byte {
    buf := pool.Get().(*bytes.Buffer)  // 为了省一次分配，代码难读了
    defer pool.Put(buf)
    // ...
}

// ✅ 先写清晰的，证明是瓶颈再优化
func (s *Service) Get() []byte {
    var buf bytes.Buffer  // 大概率够用了
    // ...
}
```
