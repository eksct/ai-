# IO 与网络编程

## 设计原理

### io.Reader / io.Writer 为什么这么设计

Go 的 IO 接口是整个标准库设计的典范——两个接口，几百个类型实现它。

```
io.Reader { Read(p []byte) (n int, err error) }
io.Writer { Write(p []byte) (n int, err error) }
```

**对比 Java：**
```
Java 的 IO 类层次：InputStream → FilterInputStream → BufferedInputStream → DataInputStream → ...
         ≈ 50 个类需要学

Go 的 IO 接口：io.Reader + io.Writer
         你只需要知道"谁实现了 Read/Write"
```

**组合而非继承：**

```go
// 想要"带缓冲的读"？包装一层：
bufReader := bufio.NewReader(file)

// 想要"边读边算哈希"？包装一层：
hashReader := io.TeeReader(file, sha256.New())

// 想要"限制读取大小"？包装一层：
limitedReader := io.LimitReader(file, 1024)
```

这就是 Go 的核心哲学：小接口 + 组合 = 强大表达力。

### Go 的 HTTP 设计

Go 的 `net/http` 包是 Go 并发模型的典型应用：

```
每个请求开一个 goroutine 处理
 → 不需要 Nginx 那样的 event loop
 → 不需要 Node.js 那样的回调
 → 写代码就像写同步程序
```

**代价：** 每个请求一个 goroutine，如果有 10000 个慢连接，就有 10000 个 goroutine。goroutine 很轻量，但也不是零成本（每个 4KB+ 栈）。高并发场景还是要用连接池和超时控制。

## 核心概念

### HTTP Server 的惯用结构

```go
// 不要把所有逻辑写在 http.HandleFunc 里
// 用 Middleware 模式分层

type Middleware func(http.Handler) http.Handler

func Logger(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}

func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                http.Error(w, "Internal Server Error", 500)
            }
        }()
        next.ServeHTTP(w, r)
    })
}

// 路由（Go 1.22+ 支持 path 参数）
mux := http.NewServeMux()
mux.HandleFunc("GET /api/images/{id}", handleGetImage)
mux.HandleFunc("POST /api/images", handleCreateImage)

// 应用中间件
handler := Recovery(Logger(mux))
http.ListenAndServe(":8080", handler)
```

## 生产实践

### HTTP 客户端超时配置

```go
// ❌ 不设置超时
client := &http.Client{}
resp, _ := client.Get("https://api.example.com")
// 如果对方服务挂了，你的 goroutine 一直不释放

// ✅ 设置所有超时
client := &http.Client{
    Timeout: 10 * time.Second,  // 整个请求最大时间（包含读取 body）
    Transport: &http.Transport{
        DialContext: (&net.Dialer{
            Timeout:   3 * time.Second,  // TCP 连接超时
            KeepAlive: 30 * time.Second,
        }).DialContext,
        TLSHandshakeTimeout:   3 * time.Second,   // TLS 握手超时
        ResponseHeaderTimeout: 5 * time.Second,   // 响应头超时
        MaxIdleConns:          100,               // 连接池最大连接数
        MaxIdleConnsPerHost:   10,                // 每个 host 最大空闲连接
    },
}
```

### 文件 I/O 注意事项

```go
// 1. 不要忽略 defer Close 的错误
f, _ := os.Open(path)
defer func() {
    if err := f.Close(); err != nil {
        log.Printf("close file error: %v", err)  // 至少打日志
    }
}()

// 2. 大文件不要用 os.ReadFile
// os.ReadFile 把整个文件加载到内存
// 处理 1GB 的图片时，内存占用就是 1GB + decode 的额外内存

// ✅ 流式处理
f, _ := os.Open(largeImage)
defer f.Close()
reader := bufio.NewReader(f)  // 默认 4KB 缓冲区
// 或
reader := bufio.NewReaderSize(f, 1024*1024)  // 1MB 缓冲区（适合大文件顺序读）
```

### 图像处理中的 IO 组合

```go
// 读取 HTTP 图片
resp, _ := http.Get("https://example.com/photo.jpg")
defer resp.Body.Close()

// 限制最大读取 10MB（防止对方传超大文件）
limitedReader := io.LimitReader(resp.Body, 10*1024*1024)

// 同时保存到文件和解码
var buf bytes.Buffer
teeReader := io.TeeReader(limitedReader, &buf)

img, _, err := image.Decode(teeReader)
if err != nil {
    return err
}

// buf 里已经存了原始数据，可以再次使用
// 不需要重新网络请求或重新读文件
```

### 文件监听防抖

```go
// 你正在做的图像管理 App 需要监听目录变化
// 但 fsnotify 的事件太频繁——复制一个文件触发 3-5 次事件

type Watcher struct {
    events    chan string
    debounced map[string]*time.Timer
    mu        sync.Mutex
}

func (w *Watcher) OnEvent(path string) {
    w.mu.Lock()
    defer w.mu.Unlock()

    if timer, ok := w.debounced[path]; ok {
        timer.Stop()  // 重置定时器
    }

    w.debounced[path] = time.AfterFunc(300*time.Millisecond, func() {
        w.events <- path  // 300ms 内没再触发才发送事件
        w.mu.Lock()
        delete(w.debounced, path)
        w.mu.Unlock()
    })
}
```
