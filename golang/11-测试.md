# 测试

## 设计原理

### Go 的测试哲学

Go 的测试是语言级的——不需要 JUnit 那样的第三方框架，不需要 mockito 那样的 mock 库。

```
Java 测试：
  用 JUnit + Mockito + AssertJ + ... 
  为了测一个方法，可能写了 5 倍于业务代码的测试代码

Go 测试：
  用 testing 包 + table-driven test
  你觉得需要 mock → 证明你代码耦合了 → 用接口解耦
```

**Go 的测试理念：** 测试不应该依赖框架，不应该做复杂的 mock。如果测不了，说明你的代码设计有问题（紧耦合），应该重构代码而不是引入 mock 框架。

## 核心概念

### Table-Driven Test

这是 Go 标准测试风格，几乎所有知名 Go 项目都用它：

```go
func TestParseImage(t *testing.T) {
    tests := []struct {
        name    string
        input   string
        want    *ImageInfo
        wantErr bool
    }{
        {
            name:    "普通 jpg",
            input:   "/photos/2024/photo.jpg",
            want:    &ImageInfo{Name: "photo", Format: "jpg"},
        },
        {
            name:    "文件名含空格",
            input:   "/photos/vacation 2024.png",
            want:    &ImageInfo{Name: "vacation 2024", Format: "png"},
        },
        {
            name:    "无扩展名",
            input:   "/photos/README",
            wantErr: true,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got, err := ParseImagePath(tt.input)
            if tt.wantErr {
                if err == nil {
                    t.Error("expected error, got nil")
                }
                return
            }
            if err != nil {
                t.Fatalf("unexpected error: %v", err)
            }
            if !reflect.DeepEqual(got, tt.want) {
                t.Errorf("ParseImagePath() = %+v, want %+v", got, tt.want)
            }
        })
    }
}
```

**好处：**
- 加一个测试用例 = 加一个结构体字段
- 测试代码没有重复
- 失败的测试用例单独命名，一眼看出哪个场景出问题

## 生产实践

### Wails 项目怎么测试

```go
// Wails App 依赖 context（调 runtime 方法用）
// 测试时没有 context，所以不能直接测 App 的 Bind 方法

// ✅ 正确做法：业务逻辑抽到独立的 service
type ImageService struct {
    // 没有 context，没有 Wails 依赖
}

func (s *ImageService) Resize(path string, w, h int) (*Image, error) {
    // 纯业务逻辑，可以单独测试
}

// App 只做胶水层
type App struct {
    service *ImageService
    ctx     context.Context
}

func (a *App) ResizeImage(path string, w, h int) *ImageResult {
    runtime.LogInfo(a.ctx, "resize: "+path)
    img, err := a.service.Resize(path, w, h)
    if err != nil {
        return &ImageResult{Error: err.Error()}
    }
    return &ImageResult{Data: img}
}

// 测试时只测 service
func TestImageService_Resize(t *testing.T) {
    svc := &ImageService{}
    tests := []struct {
        name  string
        path  string
        w, h int
    }{
        {name: "标准尺寸", path: "test.jpg", w: 800, h: 600},
        {name: "正方形", path: "test.jpg", w: 100, h: 100},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            img, err := svc.Resize(tt.path, tt.w, tt.h)
            if err != nil {
                t.Fatal(err)
            }
            if img.Bounds().Dx() != tt.w || img.Bounds().Dy() != tt.h {
                t.Errorf("尺寸不对: got %dx%d, want %dx%d",
                    img.Bounds().Dx(), img.Bounds().Dy(), tt.w, tt.h)
            }
        })
    }
}
```

### 基准测试的陷阱

```go
func BenchmarkProcess(b *testing.B) {
    // ❌ 错误：基准测试中包含了文件 IO
    for i := 0; i < b.N; i++ {
        img, _ := decodeImage("test.jpg")  // 文件 IO 会抖动
        Process(img)
    }

    // ✅ 正确：只测处理逻辑
    img, _ := decodeImage("test.jpg")
    b.ResetTimer()  // 重置计时器
    for i := 0; i < b.N; i++ {
        Process(img)
    }
}
```

### CI 中的测试配置

```yaml
# .github/workflows/test.yml
steps:
  - uses: actions/setup-go@v5
    with:
      go-version: '1.26'

  - run: go test -race -count=1 ./...    # 竞态检测 + 禁用缓存
  - run: go vet ./...                    # 静态检查
  - run: go test -coverprofile=coverage.out ./...
  - run: go tool cover -html=coverage.out -o coverage.html
```
