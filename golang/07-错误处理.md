# 错误处理

## 设计原理

### 为什么 Go 选择"返回错误"而不是"抛异常"

```
Java 的异常机制：
  try { processFile() } catch (IOException e) { ... }
  - 异常是控制流的隐式路径，调用方可能忘了 catch
  - checked exception 强制 catch，但导致很多"catch 了但不处理"的代码
  
Go 的选择：
  result, err := processFile()
  if err != nil { return err }
  - 错误是普通返回值，调用方无法忽略
  - 没有 try-catch 的嵌套开销
  - 控制流是显式的——"读代码就能看出哪些路径会出错"
```

**代价：** `if err != nil` 重复太多了。每个文件读下来，1/3 的行都是错误处理。Go 社区对此有争议，但核心团队不认为这是问题——他们说"你每看到一个 err != nil 就应该感激，因为这里可能出问题"。

### error 是值

```go
// Go 的 error 是一个接口，任何实现了 Error() string 的类型都能当 error
// 这意味着你可以创建丰富的错误类型，而不只是字符串

type error interface {
    Error() string
}

// 这点和 Java 的 Exception 不同：
// Java 的异常是对象，有类型层次、堆栈追踪、cause 链
// Go 的 error 只是一个值，堆栈追踪需要额外的包装
```

## 核心概念

### 错误包装与解包

```go
// Go 1.13 引入 %w：
// 这是 Go 版本的异常 cause 链

// 包装
func ReadConfig(path string) error {
    data, err := os.ReadFile(path)
    if err != nil {
        return fmt.Errorf("读取配置 %s 失败: %w", path, err)
    }
    // ...
}

// 解包
err := ReadConfig("/etc/app.yaml")
if errors.Is(err, os.ErrNotExist) {
    // 判断是不是"文件不存在"
    // errors.Is 会沿着包装链一直找
}

// 类型判断
var pathError *os.PathError
if errors.As(err, &pathError) {
    fmt.Println("路径:", pathError.Path)
}
```

### 自定义错误类型

```go
type ImageError struct {
    Code     string // 错误码，用于前端展示
    Message  string // 人类可读的描述
    FilePath string // 出错的图片路径
    Err      error  // 原始错误
}

func (e *ImageError) Error() string {
    return fmt.Sprintf("[%s] %s: %s", e.Code, e.Message, e.FilePath)
}

func (e *ImageError) Unwrap() error {
    return e.Err
}

// 使用
func ProcessImage(path string) error {
    file, err := os.Open(path)
    if err != nil {
        return &ImageError{
            Code:    "FILE_OPEN_ERROR",
            Message: "无法打开图片文件",
            FilePath: path,
            Err:     err,
        }
    }
    defer file.Close()
    // ...
}
```

## 生产实践

### 不要吞错误

```go
// ❌ 吞掉错误
go func() {
    doSomething()  // 如果 panic 了？整个程序挂
}()

// ✅ 处理 goroutine 中的错误
go func() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("goroutine panic: %v", r)
        }
    }()
    doSomething()
}()

// ❌ 忽略错误
json.Unmarshal(data, &result)  // data 格式不对怎么办？

// ✅ 至少打印
if err := json.Unmarshal(data, &result); err != nil {
    log.Printf("JSON 解析失败: %v", err)
    return defaultValue, err
}
```

### 错误的"归属"

```go
// 谁该处理错误？
// 原则：负责输出错误信息的一方处理错误

// 下层：提供丰富错误信息
func (s *ImageService) Process(path string) (*Image, error) {
    file, err := os.Open(path)
    if err != nil {
        return nil, fmt.Errorf("process %s: %w", path, err)
    }
    defer file.Close()
    // ...
}

// 上层：决定怎么展示错误
// Wails Bind 方法中：
func (a *App) ProcessImage(path string) *ImageResult {
    img, err := a.service.Process(path)
    if err != nil {
        runtime.LogError(a.ctx, err.Error())
        return &ImageResult{Error: err.Error()}
    }
    return &ImageResult{Data: img}
}
```

### 资源清理中的错误

```go
func saveFiles(files []string) error {
    for _, f := range files {
        if err := saveOne(f); err != nil {
            // 第一个文件出错，但已经处理了一些文件
            // 是继续还是回滚？
            // 这个问题没有标准答案，看业务要求
            return fmt.Errorf("save %s failed: %w", f, err)
        }
    }
    return nil
}

// 对于需要"全部或全不"的场景：
func saveFilesAtomic(files []string) error {
    // 先全部保存到临时位置
    saved := make([]string, 0)
    for _, f := range files {
        tmp := f + ".tmp"
        if err := saveTo(tmp, read(f)); err != nil {
            // 清理已保存的临时文件
            for _, s := range saved {
                os.Remove(s)
            }
            return err
        }
        saved = append(saved, tmp)
    }
    // 全部成功后统一移动
    for i, f := range files {
        os.Rename(saved[i], f)
    }
    return nil
}
```
