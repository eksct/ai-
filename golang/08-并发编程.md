# 并发编程

## 设计原理

### 为什么 Go 重新发明了并发模型

Go 出现之前，并发编程的三大流派：

```
1. 线程 + 锁（Java/C++）
   - 模型简单，但锁的粒度很难把控
   - 死锁、活锁、竞态问题防不胜防
   - "遇到问题加锁" → 加锁导致性能下降 → 减小锁粒度 → 死锁

2. 回调 + 事件循环（Node.js）
   - 避免了多线程问题，但回调地狱
   - 单线程无法利用多核

3. Actor 模型（Erlang）
   - 每个 actor 独立，通过消息通信
   - 但 actor 框架复杂度高，需要学习成本

Go 的选择：CSP（通信顺序进程）
   "不要通过共享内存来通信，应该通过通信来共享内存"
   本质：用 goroutine + channel 把并发问题转为数据流问题
```

### goroutine 不是线程

```go
// Java 的线程：
//   1 个线程 ≈ 1MB 栈空间
//   启动 10000 个线程 → 10GB 内存 → OOM

// Go 的 goroutine：
//   1 个 goroutine ≈ 4KB 栈空间（可动态增长）
//   启动 100000 个 goroutine → 400MB 内存 → 没问题
```

**代价：** goroutine 的调度是 Go runtime 自己做的（GMP 模型），不是操作系统线程调度。这意味着 goroutine 的上下文切换比线程快，但在 CPU 密集计算上，goroutine 无法利用线程优先级等操作系统特性。

## 核心概念

### GMP 调度模型（理解了这个，才算懂 Go 并发）

```
G = Goroutine（协程）
M = Machine（操作系统线程）
P = Processor（调度上下文）

默认 P 的数量 = GOMAXPROCS = CPU 核数

调度过程：
  1. 一个 M 必须绑定一个 P 才能执行 G
  2. G 阻塞分两种：
     - channel/mutex 阻塞 → G 从 M 摘下，M 绑定 P 去跑别的 G
     - 系统调用阻塞 → G 和 M 一起进内核，P 从 M 解绑，去窃取其他 M
  3. P 从其他 P 的队列"窃取"（steal）G 来执行（work-stealing）
  4. 核心目标：保持 M 满载，不让 P 闲着
```

**生产意义：** 理解 GMP 才能理解"为什么开 1000 个 goroutine 没事，但开 1000 个数据库连接会崩"。goroutine 阻塞 ≠ 线程阻塞。

### Channel 的类型

```go
// 无缓冲：同步通信
ch := make(chan int)
// 发送方阻塞直到有人接收
// 接收方阻塞直到有人发送
// 用于：goroutine 之间的信号同步

// 有缓冲：异步通信
ch := make(chan int, 10)
// 缓冲区满时发送方才阻塞
// 缓冲区空时接收方才阻塞
// 用于：生产-消费模型

// 只读/只写（作为参数类型约束）
func produce(ch chan<- int) { ch <- 1 }    // 只能写
func consume(ch <-chan int) { <-ch }        // 只能读
```

## 生产实践

### Worker Pool（协程池）的正确设计

```go
type WorkerPool struct {
    tasks   chan func()
    wg      sync.WaitGroup
}

func NewWorkerPool(size int) *WorkerPool {
    p := &WorkerPool{
        tasks: make(chan func(), size*10),  // 缓冲队列
    }
    p.wg.Add(size)

    for i := 0; i < size; i++ {
        go func() {
            defer p.wg.Done()
            for task := range p.tasks {
                p.safeRun(task)
            }
        }()
    }
    return p
}

func (p *WorkerPool) safeRun(task func()) {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("task panic: %v", r)
        }
    }()
    task()
}

func (p *WorkerPool) Close() {
    close(p.tasks)  // 关闭 channel，所有 worker 退出
    p.wg.Wait()     // 等待所有 worker 结束
}
```

**踩坑记录：** 某次在图像处理服务中，没有限制并发数，直接 `go processImage()` 处理 1000 张大图——瞬间打开 1000 个文件句柄，操作系统报 `too many open files`。worker pool 限制到 4 个并发后解决。

### context 的超时传递

```go
// ❌ 不做超时控制
func (a *App) ProcessLargeImage(path string) (*Image, error) {
    // 如果图片解压花 5 分钟，前端等 5 分钟
    return decodeImage(path)
}

// ✅ 带超时的 context
func (a *App) ProcessLargeImage(ctx context.Context, path string) (*Image, error) {
    // 设置 30 秒超时
    ctx, cancel := context.WithTimeout(ctx, 30*time.Second)
    defer cancel()

    resultCh := make(chan *Image, 1)
    errCh := make(chan error, 1)

    go func() {
        img, err := decodeImage(path)
        if err != nil {
            errCh <- err
            return
        }
        resultCh <- img
    }()

    select {
    case img := <-resultCh:
        return img, nil
    case err := <-errCh:
        return nil, err
    case <-ctx.Done():
        return nil, fmt.Errorf("处理超时: %w", ctx.Err())
    }
}
```

### sync.Map 适用场景

```go
// sync.Map 不是普通 map 的并发版本
// 它做了读写分离优化：读走 read 层，写走 dirty 层

// ✅ 适合 sync.Map 的场景
var cache sync.Map

func GetThumbnail(key string) ([]byte, bool) {
    v, ok := cache.Load(key)
    if ok {
        return v.([]byte), true
    }
    return nil, false
}

// ❌ 不适合 sync.Map 的场景
// 频繁写入 + 不同 key
// 此时 sync.RWMutex + 普通 map 性能更好

type SafeMap struct {
    mu   sync.RWMutex
    data map[string]int
}

func (m *SafeMap) Inc(key string) {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.data[key]++
}
```
