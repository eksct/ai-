# 包管理与模块

## 设计原理

### 从 GOPATH 到 Modules 的演变

Go 的依赖管理经历了三个阶段，理解了演变过程才能理解为什么现在是这样：

```
GOPATH 时代（Go 1.0 ~ 1.10）
  所有项目必须放在 $GOPATH/src 下
  没有版本概念，go get 拉最新代码
  项目 A：依赖 lib v1.0 → 没问题
  项目 B：依赖 lib v2.0 → 冲突了！两个项目共用同一个 GOPATH

vendor 阶段（Go 1.5 ~ 1.12）
  把依赖拷贝到项目 vendor/ 目录
  解决了"不同项目需要不同版本"的问题
  但 vendor/ 体积巨大，提交到 git 很痛苦

Modules 时代（Go 1.11+，1.16 默认）
  go.mod 声明依赖 + 版本
  go.sum 校验依赖完整性
  每个项目独立管理版本
  不再需要 GOPATH
```

**Modules 解决了什么：** 版本冲突 + 可复现构建 + 不再依赖 GOPATH。

**代价：** 私有不发布仓库的依赖管理麻烦（需要 replace 指令或 GOPRIVATE）。

### go.sum 不是锁文件

```go
// go.sum 和 npm 的 package-lock.json 不同
// package-lock.json：锁定精确版本
// go.sum：记录哈希，防止依赖被篡改

// 开发时
go mod tidy    // 更新 go.sum
// CI 中
go mod verify  // 校验依赖哈希是否匹配
```

## 核心概念

### internal 的访问控制

```go
// internal 包只能被同一 module 内的代码引用
// 这是 Go 官方的"私有包"机制

// 项目结构
image-app/
├── internal/
│   ├── processor/     // 外部项目不能引用
│   └── db/
├── main.go

// 外部项目引用 internal 编译报错：
// "use of internal package image-app/internal/processor is not allowed"
```

**生产意义：** 不需要注释说"这个包不要外部引用"——编译器直接阻止了。

## 生产实践

### 依赖版本管理

```bash
# 查看项目所有依赖
go list -m all
go list -u -m all  # 查看可升级的依赖

# 升级
go get github.com/xxx/package@latest
go get github.com/xxx/package@v1.2.3

# 清理
go mod tidy  # 删除未使用的依赖，添加缺失的
```

### replace 指令

```go
// 场景 1：私有仓库调试
// go.mod 中把远程依赖替换为本地路径
replace github.com/yourcompany/internal-lib => ../internal-lib

// 场景 2：fork 第三方包修 bug
replace github.com/someone/broken-lib => github.com/you/fixed-lib v1.0.0

// 场景 3：版本冲突时强制替换
// A 包依赖 lib v1.0，B 包依赖 lib v2.0
// 在 go.mod 中强制用 v2.0
replace lib => lib v2.0
```

### 踩坑记录

**踩坑 1：** 用了 replace 但没提交，同事拉代码后编译失败。replace 指令在 go.mod 里，要一起提交。

**踩坑 2：** 私有仓库拉取认证失败。配置 git 凭证：
```bash
git config --global url."git@github.com:yourcompany".insteadOf "https://github.com/yourcompany"
```

**踩坑 3：** `go mod tidy` 删了不该删的依赖。如果某个依赖被间接引用但不满足明显使用，tidy 会认为它无用并删除。解决方案：用 `go build` 确认能编译通过，然后提交 go.mod 和 go.sum。
