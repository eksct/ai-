# 接口

## 设计原理

### 隐式实现：Go 最颠覆性的设计

Java 的 `implements` 是显式声明："我，UserService，实现了 UserRepository 接口"。Go 的接口实现是隐式的："我的方法签名正好匹配了那个接口，所以我自动实现了它"。

```
Java 的思路：先有接口，后有实现（自上而下）
  设计阶段：interface UserRepository { User findById(int id); }
  开发阶段：class JdbcUserRepository implements UserRepository { ... }

Go 的思路：先有具体类型，后有接口（自下而上）
  开发阶段：type PostgresDB struct {}; func (db *PostgresDB) Get(id int) (*User, error)
  重构阶段：type UserStore interface { Get(id int) (*User, error) }
```

**好处：** 不需要提前规划接口。先写具体实现，用着用着发现需要抽象了再抽接口。很多 Go 项目的接口只有 1-2 个方法，小而专注。

**坏处：** 没有显式声明，阅读代码时不知道哪个类型实现了哪个接口。IDE（GoLand）可以帮你查，但看代码本身不够直观。

### 接口的"契约"本质

```go
// 接口定义的是行为契约，不是类型层次
// io.Reader 说"我能读"，io.Writer 说"我能写"
// 一个类型可以"我能读"、"我能写"、"我能关"

type Reader interface { Read(p []byte) (n int, err error) }
type Writer interface { Write(p []byte) (n int, err error) }
type Closer interface { Close() error }

// 组合接口
type ReadWriteCloser interface {
    Reader
    Writer
    Closer
}
```

## 核心概念

### 接口值的内部表示

```go
// 接口在内存中是 (type, value) 二元组
var r io.Reader = os.Stdin
// r = (type: *os.File, value: 指向标准输入)

var r io.Reader = nil
// r = (type: nil, value: nil)
// r == nil → true

var f *os.File = nil
var r io.Reader = f
// r = (type: *os.File, value: nil)
// r == nil → false !! 坑
```

**这个坑你一定会碰到：**

	```go
	func getReader() io.Reader {
	    var buf *bytes.Buffer = nil
	    return buf  // 返回的是 (type: *bytes.Buffer, value: nil)
	}
	
	r := getReader()
	if r != nil {  // true！虽然 buf 是 nil
	    // 这里会执行，然后读数据时崩溃
	}
	
	// 修复：显式返回 nil 接口值
	func getReader() io.Reader {
	    return nil  // 返回 (nil, nil)
	}
```

### 类型断言和类型 switch

```go
var v interface{} = "hello"

// 安全断言
s, ok := v.(string)
if ok { fmt.Println(s) }

// 类型 switch（比 if-else 链清晰多了）
switch val := v.(type) {
case string:
    fmt.Println("字符串:", val)
case int:
    fmt.Println("整数:", val)
case nil:
    fmt.Println("空")
default:
    fmt.Println("未知类型")
}
```

## 生产实践

### 接口放在哪

Go 社区的一个争议话题：接口定义放在"使用方"还是"实现方"？

```go
// 标准库的做法：放在使用方
// 比如 io.Reader 定义在 io 包，而不是 os 包
// os.File 实现了 Reader，但 os 包并不依赖 io

// 你的项目也应该这样做：
// 如果你的 ImageProcessor 接口被 service 包使用：
// ✅ 接口定义在 service 包
// ❌ 接口定义在 processor 包（实现方）

// service/image.go
type ImageProcessor interface {
    Resize(path string, w, h int) (*Image, error)
}

// processor/golang.go
type GoProcessor struct {}
func (p *GoProcessor) Resize(path string, w, h int) (*Image, error) {
    // ...
}
// 没有显式 import service 包
// 这样 processor 包完全没有 service 包的依赖
```

### 接受接口，返回结构体

```go
// 这是 Go 社区的黄金法则

// ✅ 接受接口：函数接收抽象行为
func SaveImage(w io.Writer, img *Image) error {
    return jpeg.Encode(w, img, nil)
}
// 可以传文件、网络连接、bytes.Buffer、压缩流...

// ✅ 返回结构体：返回具体类型，让调用方去抽象
type GoProcessor struct{}
func (g *GoProcessor) Process(img *Image) error { return nil }
func NewProcessor() *GoProcessor {
    return &GoProcessor{}
}
// 返回 *GoProcessor 而不是 ImageProcessor 接口

// 调用方直接传具体类型，单实现不需要接口
p := NewProcessor()
p.Process(img)
```

### 什么时候用接口

#### ✅ 需要接口：多个实现可替换

```go
// 单实现时直接用具体类型
type LocalStore struct{}
func (s *LocalStore) Save(id string, data []byte) error { return nil }

store := &LocalStore{}
store.Save("key", data)

// 出现第二个实现后才需要抽接口
type S3Store struct{}
func (s *S3Store) Save(id string, data []byte) error { return nil }

// 抽接口
type Store interface { Save(string, []byte) error }
func Upload(s Store, id string, data []byte) { s.Save(id, data) }

Upload(&LocalStore{}, "key", data)
Upload(&S3Store{}, "key", data)
```

#### 选型速查

```
✅ 需要用接口：
   - 多个实现可替换（数据库、缓存、存储）
   - 测试需要 mock
   - 标准库已经定义好的（io.Reader、http.Handler）

❌ 不需要用接口：
   - 只有一个实现的抽象
   - 为了"以后可能换实现"而提前抽象（YAGNI）
   - 为了"看起来面向对象"
```

**经验之谈：** 单实现时不抽接口，出现第二个实现再说。Go 的隐式实现让你任何时候抽接口都不需要改现有代码——结构体加一个方法就能满足新接口，原有代码不用动。
