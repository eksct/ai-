# Go 语言概述与环境搭建

## 设计原理

### Go 为什么诞生

Go 是 2007 年 Google 内部为解决大规模分布式系统开发痛点而设计的。C++ 编译太慢、依赖管理混乱、并发模型复杂。Java 太重、虚拟机层带来额外复杂度、部署麻烦。Python/Ruby 性能不够、类型不安全。

**核心目标：**
- 编译快（秒级，拒绝 C++ 那样的半小时编译）
- 并发简单（goroutine 比线程池/回调/异步 await 都更直观）
- 部署简单（静态链接，单二进制，无 JVM 依赖）
- 开发效率高（极简语法，内置工具链）

### Go 的设计权衡

```
解决的问题               付出的代价
──────────────────────────────────
编译快                   泛型直到 1.18 才加入（编译速度和泛型难两全）
部署简单（单二进制）       包体积较大（包含 runtime，约 2-7MB）
并发简单（goroutine）     无法精确控制线程优先级、调度策略
代码简洁                 表达能力受限（没有继承、没有重载、没有异常）
没有历史包袱              放弃向后兼容某些特性（go get 的演变）
```

### 和 Java 的对比

| 对比项 | Go | Java |
|--------|-----|------|
| 编译 | 直接编译为二进制 | 编译为字节码，JVM 运行 |
| 并发模型 | goroutine + channel（CSP） | 线程 + 线程池 + 锁 |
| 部署 | 单二进制，无依赖 | 需要 JRE，jar 包 |
| 泛型 | 1.18 加入，约束有限 | 从 5.0 开始，完善 |
| 异常 | error 返回值 | try-catch 异常机制 |
| 语法 | 25 个关键字 | ~50 个关键字 |
| 包管理 | go mod（官方） | Maven/Gradle（第三方） |
| 学习曲线 | 语法简单，并发理解难 | 语法复杂，生态庞大 |

**踩坑记录：** 从 Java 转 Go 最容易犯的错不是学不会 goroutine，而是"用 Java 的思维写 Go"——硬把 Go 代码写成继承层次结构、大量用接口抽象没实际收益、习惯性抛 panic 而不是返回 error。

## 核心概念

### 环境变量

```bash
# 不配也能用，但建议显式设置
go env -w GOPROXY=https://goproxy.cn,https://goproxy.io,direct  # 国内必配
go env -w GO111MODULE=on                                         # 确保 module 模式
```

**GO111MODULE 的演变：**
- Go 1.11~1.15：默认 auto（项目有 go.mod 就用 module，否则用 GOPATH）
- Go 1.16+：默认 on（不再支持 GOPATH 模式）
- 如果你还在配 GOPATH，说明你学的教程至少是 5 年前的

### Module 设计原理

```
Java 的依赖管理思路：
  每个 jar 包是一个 module，通过 Maven/Gradle 解决版本冲突。
  但"传递依赖"和"版本冲突"永远在打架。

Go module 的思路：
  不解决版本冲突——只允许一个项目中每个依赖有一个版本。
  需要不同版本？你自己 fork 或 vendor。
```

**经验之谈：** Go 的"一个版本"哲学在实践中会遇到问题——A 包依赖 v1，B 包依赖 v2，这时候就得靠 replace 指令手动解决。看起来粗糙，但实际工作流中比 Maven 的版本仲裁要可预测得多。

## 生产实践

### 选型建议

**什么项目适合 Go：**
- CLI 工具（跨平台单二进制，开箱即用）
- API 网关 / 代理服务（高吞吐、低延迟）
- 中间件 / 基础设施（Docker, K8s, etcd, Prometheus 都是 Go 写的）
- 桌面应用（Wails 方案）
- 图像/音视频处理服务

**什么项目不建议 Go：**
- 复杂业务系统（缺少继承、重载，大型业务抽象表达力弱）
- 数据科学/AI（生态远不如 Python）
- 前端/移动端（不是 Go 的领域）

### Wails 项目结构经验

```
image-app/
├── main.go            # 入口，尽量薄（只调 wails.Run）
├── app.go             # App struct + Bind 方法
├── internal/
│   ├── processor/     # 图像处理核心（可独立测试，不依赖 Wails）
│   ├── metadata/      # EXIF 读取
│   └── cache/         # 缩略图缓存
├── frontend/          # 前端（Vue/React）
└── build/             # Wails 构建配置
```

**教训：** 不要把所有逻辑塞 App struct。App 应该是"胶水层"，业务逻辑抽到 `internal/`。这样测试不用启动 Wails，单元测试秒跑。

### Go 版本选择

| 版本 | 状态 | 建议 |
|------|------|------|
| 1.21 | 当前版本 | 新项目用 |

| 1.18~1.19 | 已过时 | 除非有兼容问题 |
| 1.17 以下 | 过期 | 强制升级 |
