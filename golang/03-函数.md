# 函数

## 设计原理

### 为什么 Go 的函数这样设计

**多返回值：** Go 没有异常机制，也没有三元运算符。用多返回值返回 `(result, error)` 是 Go 最核心的错误处理模式。

和 Java 对比：
```
Java: 
  User user = findUser(id);       // 可能抛异常，也可能返回 null
  // 调用方可能忘记 try-catch，也可能忘记判空

Go:
  user, err := findUser(id)       // 强制处理 error
  if err != nil { return err }
  // user 不可能为 nil（或者你显式返回 nil）
```

这是 Go 的核心理念：错误是返回值，不是控制流。

**defer：** Go 没有 Java 的 try-finally，也不推荐 Java 那种 `try(Resource r = ...)` 语法。defer 的语义是"函数返回前必执行"，比 finally 更灵活——你不必在函数开头就规划好释放逻辑，可以在获取资源后立即写 defer。

## 核心概念

### 函数是一等公民

```go
// Go 的函数可以赋值变量、传递参数、作为返回值
// 这是实现函数式编程和回调的基础

type Handler func(w http.ResponseWriter, r *http.Request)
type Middleware func(Handler) Handler

// 中间件链
func Logging(next Handler) Handler {
    return func(w http.ResponseWriter, r *http.Request) {
        log.Printf("%s %s", r.Method, r.URL)
        next(w, r)
    }
}

func Auth(next Handler) Handler {
    return func(w http.ResponseWriter, r *http.Request) {
        if !isAuthenticated(r) {
            http.Error(w, "unauthorized", http.StatusUnauthorized)
            return
        }
        next(w, r)
    }
}

// 使用
handler := Logging(Auth(finalHandler))
```

### 闭包的捕获机制

```go
// ❌ Bug: 闭包捕获的是变量引用，不是值
for i := 0; i < 3; i++ {
    defer func() {
        fmt.Println(i)  // 循环结束时 i=3，全部打印 3
    }()
}

// ✅ 修复 1：传参（值传递）
for i := 0; i < 3; i++ {
    defer func(n int) {
        fmt.Println(n)  // 2, 1, 0
    }(i)
}

// ✅ 修复 2：局部变量遮蔽
for i := 0; i < 3; i++ {
    i := i  // 循环内每次创建新变量
    defer func() {
        fmt.Println(i)  // 2, 1, 0
    }()
}
```

**经验：** Go 1.22+ 循环变量不再被闭包捕获问题困扰（1.22 修复了每次迭代复用同一变量的行为）。如果你还在用 1.22 之前的版本，需要手动 `i := i` 或传参。

## 生产实践

### defer 的最佳实践

```go
// 1. 获取资源后立即 defer
f, err := os.Open(file)
if err != nil { return err }
defer f.Close()

// 2. 锁操作
mu.Lock()
defer mu.Unlock()

// 3. 函数耗时统计（闭包捕获外部变量）
func process() {
    start := time.Now()
    defer func() {
        log.Printf("process 耗时: %v", time.Since(start))
    }()
    // ...
}

// 4. 不要和返回值一起用（容易犯错）
// 错误的 defer 修改返回值
func getValue() (result int) {
    defer func() {
        result = 42  // 改了返回值
    }()
    return 0
}
// getValue() 返回 42，不是 0
```

### defer 性能

```go
// defer 有微小开销（一次函数调用 + 栈操作）
// 每秒几百万次调用的热点函数不建议用
// 一般业务代码完全不用在意
```

### init 函数该不该用

```go
// init 适合的场合：
// 1. 包级别的变量初始化，且不需要外部参数
// 2. 注册驱动（数据库、编解码器）

// init 不要做的：
// 1. 读配置文件（因为 init 没有入参，依赖全局变量，测试困难）
// 2. 连数据库（同理）
// 3. 启动 goroutine

// ✅ 好的 init
func init() {
    image.RegisterFormat("webp", "RIFF????WEBPVP8", webp.Decode, webp.DecodeConfig)
}

// ❌ 不好的 init
var db *sql.DB
func init() {
    var err error
    db, err = sql.Open("sqlite", config.DBPath)
    if err != nil {
        log.Fatal(err)  // 测试时也会执行，且无法换库
    }
}
```

### panic/recover 该不该用

```go
// recover 不是 Go 版本的 try-catch
// 它的设计意图是：在 goroutine 顶层捕获未知 panic，防止进程崩溃

// ✅ 该用的地方
go func() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("goroutine panic: %v", r)
        }
    }()
    // 可能会 panic 的代码
}()

// ❌ 不该用的地方
func divide(a, b int) int {
    defer func() {
        if r := recover(); r != nil {
            // 用 panic 做业务逻辑控制——不要这样做
        }
    }()
    return a / b
}
// 应该用：if b == 0 { return 0, errors.New("divide by zero") }
```
