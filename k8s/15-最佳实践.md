# Kubernetes 最佳实践

## 一、资源管理

### 1. 始终设置 Requests 和 Limits

```yaml
# ✅ 生产必须设置
resources:
  requests:
    cpu: 500m
    memory: 512Mi
  limits:
    cpu: "1"
    memory: 1Gi
```

**为什么？**
- 没有 requests → 调度器无法合理分配资源
- 没有 limits → 单个 Pod 可能耗尽节点资源
- 两者差距大 → 节点过载风险

### 2. 使用不同的 QoS 等级

```yaml
# Guaranteed（关键服务）
resources:
  requests:
    cpu: "1"
    memory: 1Gi
  limits:
    cpu: "1"
    memory: 1Gi

# Burstable（普通服务）
resources:
  requests:
    cpu: 200m
    memory: 256Mi
  limits:
    cpu: "1"
    memory: 1Gi

# BestEffort（高可压缩任务，不设置 resources）
# 无资源声明的 Pod（不推荐）
```

### 3. 合理设置 HPA

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-hpa
spec:
  # 给 HPA 足够的缓冲时间避免震荡
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
        - type: Percent
          value: 100
          periodSeconds: 15
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

## 二、部署策略

### 滚动更新配置

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0      # 零停机更新
      maxSurge: 1            # 每次多启动一个
  minReadySeconds: 30        # 等待 30s 确认 Pod 就绪
  revisionHistoryLimit: 5    # 保留 5 个旧版本用于回滚
```

### 蓝绿部署

```bash
# 创建新版本的 Service（标签不同）
kubectl apply -f deployment-v2.yaml

# 验证新版本正常后，切换 Service 标签
kubectl patch service my-app -p '{"spec":{"selector":{"version":"v2"}}}'

# 确认无误后删除旧版本
kubectl delete deployment my-app-v1
```

### 金丝雀发布

```bash
# 使用 Service Mesh（Istio）
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
    - my-app
  http:
    - route:
        - destination:
            host: my-app
            subset: v1
          weight: 90
        - destination:
            host: my-app
            subset: v2
          weight: 10
```

## 三、高可用设计

### 1. 跨可用区部署

```yaml
spec:
  topologySpreadConstraints:
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule
      labelSelector:
        matchLabels:
          app: my-app
  affinity:
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: my-app
            topologyKey: kubernetes.io/hostname
```

### 2. PDB（Pod 中断预算）

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: app-pdb
spec:
  minAvailable: 2          # 最少保持 2 个可用
  selector:
    matchLabels:
      app: my-app
```

### 3. 多副本 + 反亲和性

```yaml
spec:
  replicas: 3
  # 至少跨 3 个节点、2 个可用区
```

## 四、安全实践

### 1. Pod Security

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: nginx
      securityContext:
        allowPrivilegeEscalation: false
        privileged: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
          add: ["NET_BIND_SERVICE"]
```

### 2. RBAC 最小权限

```yaml
# ❌ 不要这样做
kind: RoleBinding
subjects:
  - kind: ServiceAccount
    name: my-app
roleRef:
  kind: ClusterRole
  name: cluster-admin       # 太大了！

# ✅ 应该这样做：只给需要的权限
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: my-app-role
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list"]
```

### 3. NetworkPolicy

```yaml
# 默认拒绝所有入站
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny
spec:
  podSelector: {}
  policyTypes:
    - Ingress
---
# 只允许指定来源访问
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - port: 8080
```

  memory: 2Gi

  cpu: "1"
  memory: 2Gi
evictionHard:
  memory.available: "500Mi"
  nodefs.available: "10%"

## 七、成本优化

### 1. 合理设置资源请求

```bash
# 使用 VPA 推荐资源配置
# 或使用 Kubecost / Goldilocks 分析实际利用率
```

```yaml
# 根据历史使用数据调整
resources:
  requests:
    cpu: 200m      # 降低到实际使用量的 110%
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
```

### 2. 使用 Spot 实例（节省 60-80%）

```yaml
# AWS 使用 Spot 实例
spec:
  template:
    spec:
      nodeSelector:
        lifecycle: Ec2Spot
      tolerations:
        - key: "spotInstance"
          operator: "Exists"
```

### 3. 自动缩容到 0

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: cron-worker
spec:
  scaleTargetRef:
    name: cron-worker
  minReplicaCount: 0                # 无工作时缩到 0
  triggers:
    - type: cron
      metadata:
        timezone: Asia/Shanghai
        start: 0 9 * * *
        end: 0 18 * * *
        desiredReplicas: "10"
```
