# Kubernetes 网络原理

## K8s 网络模型

K8s 对网络提出 4 个核心要求：

1. **Pod 间可以直接通信**（无需 NAT），跨节点 Pod 互通
2. **节点与 Pod 可以直接通信**
3. **Pod 看到自己的 IP 与他人看到的相同**
4. **所有容器共享 Pod 的网络命名空间**

### 网络模型示意图

```
Node 1                              Node 2
┌──────────────────┐               ┌──────────────────┐
│ Pod A: 10.1.1.2  │               │ Pod C: 10.1.3.2  │
│   ┌─────┐        │               │   ┌─────┐        │
│   │c1:80│─┐      │               │   │c1:80│─┐      │
│   └─────┘ │      │               │   └─────┘ │      │
│           │eth0  │               │           │eth0  │
│ Pod B: 10.1.1.3 │───────────────│ Pod D: 10.1.3.3  │
│   ┌─────┐       │               │   ┌─────┐        │
│   │c1:80│─┐     │               │   │c1:80│─┐      │
│   └─────┘ │     │               │   └─────┘ │      │
│           │eth0 │               │           │eth0  │
│           │     │               │           │      │
│           │     │               │           │      │
│ CNI Bridge│     │               │ CNI Bridge│      │
└───────────┴─────┘               └───────────┴──────┘
```

## CNI（Container Network Interface）

CNI 是 K8s 网络的插件标准，由 kubelet 调用：

```
Pod 创建 → kubelet → 调用 CNI 插件 → 分配网络 → 完成
```

### 主流 CNI 插件对比

| 插件 | 网络模式 | 性能 | 特性 |
|------|----------|------|------|
| **Calico** | BGP/VXLAN | ⭐⭐⭐⭐ | NetworkPolicy, 安全策略丰富 |
| **Flannel** | VXLAN/Host-GW | ⭐⭐⭐ | 简单，适用于小集群 |
| **Cilium** | eBPF | ⭐⭐⭐⭐⭐ | 高性能，可替代 kube-proxy |
| **Weave** | VXLAN/FastDP | ⭐⭐⭐ | 内置 DNS, 加密 |
| **Kube-OVN** | OVN/OVS | ⭐⭐⭐⭐ | 企业级，支持 QoS |
| **Antrea** | Open vSwitch | ⭐⭐⭐⭐ | VMware 出品 |

## Flannel

Flannel 是最简单的 CNI，通过 VXLAN 隧道实现跨节点通信：

```bash
# 安装 Flannel
kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml

# Flannel 配置（ConfigMap）
kubectl -n kube-system get cm kube-flannel-cfg -o yaml
```

**网络流程：**
```
Pod A (10.1.1.2) → flannel.1 → VXLAN → Node B → flannel.1 → Pod B (10.1.2.3)
                                     ↑
                             物理网络（UDP 封装）
```

## Calico

Calico 是生产环境中**最常用**的 CNI 插件，支持 BGP 直接路由和 NetworkPolicy。

### 安装

```bash
# 安装 Calico（推荐使用 Tigera Operator）
kubectl create -f https://raw.githubusercontent.com/projectcalico/calico/v3.27/manifests/tigera-operator.yaml

# 通过 CR 配置
cat <<EOF | kubectl apply -f -
apiVersion: operator.tigera.io/v1
kind: Installation
metadata:
  name: default
spec:
  calicoNetwork:
    ipPools:
      - cidr: 10.244.0.0/16
        encapsulation: VXLANCrossSubnet
        natOutgoing: true
    nodeAddressAutodetectionV4:
      interface: "eth0"
EOF
```

### BGP 模式 vs VXLAN 模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **BGP** | 直接路由，无封装 | 同一二层网络，高性能 |
| **VXLAN** | Overlay 隧道 | 跨子网/云环境 |
| **VXLANCrossSubnet** | 同子网 BGP，跨子网 VXLAN | 混合场景（推荐） |

### NetworkPolicy 实现

```yaml
apiVersion: projectcalico.org/v3
kind: GlobalNetworkPolicy
metadata:
  name: default-deny
spec:
  selector: all()
  types:
    - Ingress
    - Egress
---
# Calico 全局策略比 K8s NetworkPolicy 更强：
apiVersion: projectcalico.org/v3
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: production
spec:
  selector: app == 'frontend'
  types:
    - Egress
  egress:
    - action: Allow
      protocol: UDP
      destination:
        ports:
          - 53
    - action: Allow
      protocol: TCP
      destination:
        ports:
          - 53
```

## Cilium（eBPF 方案）

Cilium 基于 eBPF 技术，性能极高，**可完全替代 kube-proxy**：

```bash
# 安装 Cilium
helm repo add cilium https://helm.cilium.io/
helm install cilium cilium/cilium \
  --namespace kube-system \
  --set kubeProxyReplacement=true \
  --set hubble.enabled=true \
  --set hubble.relay.enabled=true \
  --set hubble.ui.enabled=true
```

**Cilium 的优势：**
- 无 iptables 性能损耗（eBPF 直接在内核执行）
- Service 负载均衡性能高
- 可观测性强（Hubble）
- 支持 L7 NetworkPolicy
- 支持加密通信（WireGuard）

```yaml
# Cilium L7 NetworkPolicy（基于 HTTP 方法/路径）
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: l7-policy
spec:
  endpointSelector:
    matchLabels:
      app: backend
  ingress:
    - fromEndpoints:
        - matchLabels:
            app: frontend
      toPorts:
        - ports:
            - port: "8080"
              protocol: TCP
          rules:
            http:
              - method: "GET"
                path: "/api/v1/public"
              - method: "POST"
                path: "/api/v1/data"
```

## kube-proxy 工作原理

kube-proxy 负责 Service 的负载均衡实现：

### iptables 模式（默认）

```
Service: my-service (10.96.12.34:80)
  ↓
PREROUTING → KUBE-SERVICES → KUBE-SVC-XXXX
  ↓
KUBE-SVC-XXXX → 随机选择一条规则
  ├── KUBE-SEP-AAAA → Pod 1 (10.1.1.2:80)
  ├── KUBE-SEP-BBBB → Pod 2 (10.1.1.3:80)
  └── KUBE-SEP-CCCC → Pod 3 (10.1.1.4:80)
```

### IPVS 模式（推荐用于大规模集群）

```bash
# 启用 IPVS 模式
kubectl edit configmap kube-proxy -n kube-system
# ipvs:
#   scheduler: rr    # 轮询

# 支持调度算法：rr / lc / dh / sh / sed / nq
```

## DNS（服务发现）

K8s DNS 基于 CoreDNS，是集群内部服务发现的核心：

```bash
# 查看 CoreDNS
kubectl -n kube-system get pods -l k8s-app=kube-dns
kubectl -n kube-system get configmap coredns -o yaml
```

### DNS 记录格式

```
<service>.<namespace>.svc.cluster.local

# 示例
my-service.default.svc.cluster.local  → ClusterIP
my-service.default.svc.cluster.local  → Pod IP 列表（Headless Service）
```

### Pod DNS 配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: dns-demo
spec:
  dnsPolicy: ClusterFirst               # 默认，先查集群 DNS
  # dnsPolicy: Default                  # 使用节点 DNS
  # dnsPolicy: None                     # 完全自定义
  dnsConfig:
    nameservers:
      - 8.8.8.8
      - 223.5.5.5
    searches:
      - default.svc.cluster.local
      - svc.cluster.local
      - cluster.local
    options:
      - name: ndots
        value: "2"
      - name: timeout
        value: "3"
  containers:
    - name: app
      image: nginx
```

## 网络排查工具

### 部署网络调试 Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: net-debug
spec:
  containers:
    - name: netshoot
      image: nicolaka/netshoot:latest
      command: ["sleep", "3600"]
  hostNetwork: true           # 使用宿主机网络栈
---
# 不共享宿主机网络的版本
apiVersion: v1
kind: Pod
metadata:
  name: net-debug-pod
spec:
  containers:
    - name: netshoot
      image: nicolaka/netshoot:latest
      command: ["sleep", "3600"]
```

```bash
# 进入调试 Pod
kubectl exec -it net-debug -- bash

# 常用排查命令
ip addr                                    # 查看网络接口
ip route                                   # 查看路由表
arp -a                                     # ARP 缓存
ping 10.96.0.10                            # 测试 CoreDNS
nslookup kubernetes.default.svc.cluster.local  # DNS 解析
curl http://my-service:8080                # HTTP 测试
tcpdump -i any port 80                    # 抓包
ss -tunap                                  # 查看连接
```

### 常见网络问题排查

```bash
# 1. DNS 解析失败
kubectl exec -it pod -- nslookup kubernetes.default
kubectl logs -n kube-system -l k8s-app=kube-dns

# 2. Pod 无法互访
kubectl get endpoints <service>            # 检查 Endpoints
kubectl describe pod <pod> | grep -A5 "IP"  # 检查 Pod IP
kubectl exec -it pod -- ping <target-ip>

# 3. 外部访问失败
kubectl get svc -A                         # 检查 Service 类型
kubectl describe svc <service>             # 检查端口映射

# 4. NetworkPolicy 问题
kubectl get networkpolicy -A               # 查看所有策略
kubectl describe networkpolicy <name>

# 5. 节点间网络
ssh node-01 -- ping node-02                # 测试节点连通性
ssh node-01 -- curl <pod-ip>:<port>        # 测试 Pod 连通性
```

## 网络性能优化

```yaml
# 1. 启用 IPVS 模式（大集群）
# 2. 选择高性能 CNI（Cilium > Calico > Flannel）
# 3. 调整内核参数
apiVersion: v1
kind: ConfigMap
metadata:
  name: tuned-sysctl
data:
  90-kubernetes.conf: |
    net.core.somaxconn = 32768
    net.ipv4.tcp_max_syn_backlog = 16384
    net.ipv4.ip_local_port_range = 1024 65000
    net.ipv4.tcp_tw_reuse = 1
    net.core.rmem_max = 16777216
    net.core.wmem_max = 16777216
---
# 4. 本地流量优先保留在节点内
service.spec.externalTrafficPolicy: Local
```
