# 调度与亲和性

## 调度流程

```
Pod 创建 → API Server 写入 etcd
         → Scheduler 监听未调度 Pod
         → Filtering（过滤不可用节点）
         → Scoring（对可用节点打分）
         → Binding（绑定到最优节点）
         → kubelet 在该节点启动 Pod
```

### 调度阶段

```
Filtering（预选）:
  - 节点是否满足 Pod 的资源请求
  - 节点端口是否冲突
  - 节点是否有对应的 NodeSelector/Label
  - 污点容忍检查
  - 亲和性规则检查

Scoring（优选）:
  - 节点剩余资源（最常用）
  - Pod 分布均匀性
  - 节点亲和性权重
  - 自定义调度器扩展
```

## NodeSelector（节点选择器）

最简单的节点选择方式：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  nodeSelector:
    gpu: "true"              # 只调度到有 gpu=true 标签的节点
    disktype: ssd
  containers:
    - name: cuda
      image: nvidia/cuda:12.0
```

```bash
# 给节点打标签
kubectl label node node-01 gpu=true
kubectl label node node-01 disktype=ssd

# 查看节点标签
kubectl get nodes --show-labels
```

**局限：** 只支持"与"关系，不支持"或"、"非"等复杂逻辑。

## 节点亲和性（Node Affinity）

比 NodeSelector 更灵活的节点选择方式：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-affinity-demo
spec:
  affinity:
    nodeAffinity:
      # 必须满足的条件（硬性要求）
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                  - us-east-1a
                  - us-east-1b
              - key: gpu
                operator: Exists

      # 优先满足的条件（软性要求）
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 80
          preference:
            matchExpressions:
              - key: disktype
                operator: In
                values:
                  - ssd
        - weight: 20
          preference:
            matchExpressions:
              - key: environment
                operator: In
                values:
                  - production
  containers:
    - name: app
      image: nginx
```

### 操作符

| 操作符 | 行为 |
|--------|------|
| `In` | 标签值在指定集合中 |
| `NotIn` | 标签值不在指定集合中 |
| `Exists` | 存在该标签（无论值是什么） |
| `DoesNotExist` | 不存在该标签 |
| `Gt` | 标签值大于指定值（字符串比较） |
| `Lt` | 标签值小于指定值（字符串比较） |

## Pod 亲和性（Pod Affinity / Anti-Affinity）

控制 Pod 之间的调度关系：

### Pod 亲和性（同位置调度）

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cache-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: cache
  template:
    metadata:
      labels:
        app: cache
    spec:
      affinity:
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchExpressions:
                  - key: app
                    operator: In
                    values:
                      - frontend
              topologyKey: topology.kubernetes.io/zone
              # 与 app=frontend 的 Pod 在同一可用区
      containers:
        - name: cache
          image: redis:7-alpine
```

### Pod 反亲和性（分散调度）

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-deployment
spec:
  replicas: 5
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchExpressions:
                    - key: app
                      operator: In
                      values:
                        - web
                topologyKey: kubernetes.io/hostname
                # 尽量让 web Pod 不分布在同一个节点上
      containers:
        - name: web
          image: nginx:alpine
```

### topologyKey 说明

| topologyKey | 范围 |
|-------------|------|
| `kubernetes.io/hostname` | 节点级别 |
| `topology.kubernetes.io/zone` | 可用区级别 |
| `topology.kubernetes.io/region` | 地域级别 |
| `failure-domain.beta.kubernetes.io/zone` | 旧版可用区（已弃用） |

## 污点（Taint）和容忍（Toleration）

污点标记节点拒绝特定 Pod 调度，容忍是 Pod 声明的"我能忍受这个污点"。

### 节点污点

```bash
# 添加污点
kubectl taint nodes node-01 key=value:NoSchedule
kubectl taint nodes node-01 gpu=true:NoSchedule
kubectl taint nodes node-01 dedicated=ml:NoSchedule

# 查看污点
kubectl describe node node-01 | grep Taints

# 删除污点
kubectl taint nodes node-01 key=value:NoSchedule-
kubectl taint nodes node-01 gpu=true:NoSchedule-
```

### 污点效果（Effect）

| 效果 | 行为 |
|------|------|
| `NoSchedule` | 不容忍的 Pod 不能调度到该节点（已运行的 Pod 不受影响） |
| `PreferNoSchedule` | 尽量不调度（尽量规避，但不强制） |
| `NoExecute` | 不容忍的 Pod 被驱逐，已运行的也会被驱离 |

### Pod 容忍

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: taint-toleration-demo
spec:
  tolerations:
    - key: "gpu"
      operator: "Equal"
      value: "true"
      effect: "NoSchedule"
    - key: "dedicated"
      operator: "Exists"                # 只要存在该键即可
      effect: "NoSchedule"
    - key: "node.kubernetes.io/not-ready"
      operator: "Exists"
      effect: "NoExecute"
      tolerationSeconds: 300            # 容忍 300 秒后驱逐
  containers:
    - name: app
      image: nginx
```

### 内置污点（节点故障时自动添加）

| 污点 | 条件 |
|------|------|
| `node.kubernetes.io/not-ready` | 节点 NotReady |
| `node.kubernetes.io/unreachable` | 节点不可达 |
| `node.kubernetes.io/out-of-disk` | 磁盘不足 |
| `node.kubernetes.io/memory-pressure` | 内存压力 |
| `node.kubernetes.io/disk-pressure` | 磁盘压力 |
| `node.kubernetes.io/network-unavailable` | 网络不可用 |
| `node.kubernetes.io/unschedulable` | 节点不可调度 |
| `node.cloudprovider.kubernetes.io/uninitialized` | 云提供商初始化中 |

## Taint 的实战应用

### 专用 GPU 节点

```bash
# 给 GPU 节点打污点
kubectl taint nodes gpu-node-01 gpu=true:NoSchedule

# 只有声明了容忍的 ML 任务才能调度到 GPU 节点
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: ml-training-job
spec:
  tolerations:
    - key: "gpu"
      operator: "Equal"
      value: "true"
      effect: "NoSchedule"
  nodeSelector:
    gpu: "true"
  containers:
    - name: trainer
      image: tensorflow/tensorflow:latest-gpu
      resources:
        limits:
          nvidia.com/gpu: 1
```

### 节点运维

```bash
# 准备维护节点：驱逐 Pod 并禁止调度
kubectl drain node-01 --ignore-daemonsets --delete-emptydir-data

# 维护完成后恢复
kubectl uncordon node-01
```

## 调度器配置（Scheduler Configuration）

### 多个调度器

集群可以运行多个调度器，Pod 可以选择使用哪个：

```bash
# 部署自定义调度器
kubectl apply -f my-scheduler.yaml
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: custom-scheduled-pod
spec:
  schedulerName: my-scheduler    # 指定使用自定义调度器
  containers:
    - name: app
      image: nginx
```

### 调度器扩展点

K8s 调度器支持通过以下方式扩展：

1. **Scheduler Framework**：编写插件（Go）
2. **Webhook**：通过外部 HTTP 服务扩展
3. **Descheduler**：重新平衡集群负载

## Pod Topology Spread Constraints（拓扑分布约束）

精细控制 Pod 在集群中的分布：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-spread
spec:
  replicas: 9
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      topologySpreadConstraints:
        - maxSkew: 1                    # 最大不平衡数
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: web
        - maxSkew: 2
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels:
              app: web
      containers:
        - name: web
          image: nginx
```

**whenUnsatisfiable 选项：**
- `DoNotSchedule`：无法满足时禁止调度
- `ScheduleAnyway`：尽量满足，不能满足就算了

## 实战组合示例

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: production-app
spec:
  replicas: 10
  selector:
    matchLabels:
      app: production-app
  template:
    metadata:
      labels:
        app: production-app
    spec:
      # 1. 节点亲和性：优先 SSD 节点
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
              - matchExpressions:
                  - key: environment
                    operator: In
                    values:
                      - production
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              preference:
                matchExpressions:
                  - key: disktype
                    operator: In
                    values:
                      - ssd
        # 2. Pod 反亲和性：分散在不同节点
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchExpressions:
                    - key: app
                      operator: In
                      values:
                        - production-app
                topologyKey: kubernetes.io/hostname
      # 3. 拓扑分布约束：均匀分布在可用区
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: production-app
      # 4. 容忍系统污点
      tolerations:
        - operator: "Exists"
          effect: "NoExecute"
          tolerationSeconds: 300
      containers:
        - name: app
          image: nginx:alpine
```

## 调度决策速查表

```
需求                          → 方案
────────────────────────────────────────────────────
调度到特定标签的节点           → nodeSelector
调度到标签匹配的节点（复杂条件） → NodeAffinity required
优先调度到某类节点             → NodeAffinity preferred
不让两个 Pod 在同一节点        → PodAntiAffinity
让某类 Pod 尽量在一起          → PodAffinity
不让 Pod 调度到某节点          → node Taint + Pod Toleration
均匀分布 Pod                   → TopologySpreadConstraints
独占节点                      → Taint: NoSchedule
维护节点（驱逐 Pod）           → kubectl drain
```
