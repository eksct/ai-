# 接口文档

## 概述

不写文档的前后端协作就是互相甩锅。但手写 Word 文档是反人类行为——更新不及时、没人维护、和代码脱节。

**生产方案：** Knife4j（国产 Swagger 增强版），注解自动生成文档，代码即文档。

## Knife4j 集成

### 依赖

```xml
<dependency>
    <groupId>com.github.xiaoymin</groupId>
    <artifactId>knife4j-openapi3-jakarta-spring-boot-starter</artifactId>
    <version>4.5.0</version>
</dependency>
```

### 配置

```java
@Configuration
public class Knife4jConfig {
    @Bean
    public OpenAPI customOpenAPI() {
        return new OpenAPI()
                .info(new Info()
                        .title("订单服务 API")
                        .version("1.0.0")
                        .description("订单服务接口文档")
                        .contact(new Contact().name("后端团队").email("dev@example.com")))
                .addSecurityItem(new SecurityRequirement().addList("Bearer"))
                .components(new Components()
                        .addSecuritySchemes("Bearer", new SecurityScheme()
                                .name("Authorization")
                                .type(SecurityScheme.Type.HTTP)
                                .scheme("bearer")
                                .bearerFormat("JWT")));
    }
}
```

```yaml
springdoc:
  swagger-ui:
    path: /swagger-ui.html
  api-docs:
    path: /v3/api-docs

knife4j:
  enable: true
  setting:
    language: zh_cn
    enable-after-script: false
```

## 接口文档规范

### Controller 注解

```java
@Tag(name = "用户管理", description = "用户 CRUD 操作")
@RestController
@RequestMapping("/api/users")
public class UserController {

    @Operation(summary = "用户列表", description = "分页查询用户列表")
    @Parameters({
        @Parameter(name = "page", description = "页码", required = true),
        @Parameter(name = "size", description = "每页条数", required = true),
        @Parameter(name = "status", description = "状态筛选")
    })
    @GetMapping
    public ApiResult<PageResult<UserVO>> listUsers(
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "20") int size,
            @RequestParam(required = false) Integer status) {
        return ApiResult.success(userService.list(page, size, status));
    }

    @Operation(summary = "用户详情")
    @GetMapping("/{id}")
    public ApiResult<UserVO> getUser(@Parameter(description = "用户 ID") @PathVariable Long id) {
        return ApiResult.success(userService.getById(id));
    }

    @Operation(summary = "创建用户")
    @PostMapping
    public ApiResult<Long> createUser(@Valid @RequestBody @Parameter(description = "用户信息")
                                          UserCreateReq req) {
        return ApiResult.success(userService.create(req));
    }
}
```

### 请求体对象注解

```java
@Data
@Schema(description = "创建用户请求")
public class UserCreateReq {

    @Schema(description = "用户名", example = "alice")
    @NotBlank(message = "用户名不能为空")
    private String username;

    @Schema(description = "密码", example = "Abc12345")
    @NotBlank
    private String password;

    @Schema(description = "邮箱", example = "alice@example.com")
    @Email
    private String email;

    @Schema(description = "角色ID列表", example = "[1, 2]")
    private List<Long> roleIds;
}
```

### 响应体注解

```java
@Data
@Schema(description = "用户信息")
public class UserVO {
    @Schema(description = "用户ID", example = "123456")
    private Long id;

    @Schema(description = "用户名", example = "alice")
    private String username;

    @Schema(description = "邮箱", example = "alice@example.com")
    private String email;

    @Schema(description = "创建时间", example = "2024-01-01 12:00:00")
    private LocalDateTime createTime;

    @Schema(description = "状态 0=启用 1=禁用", example = "0")
    private Integer status;
}
```

## 接口定义规范

### 团队必须统一的规则

```
1. URL 路径用复数名词
   ✅ /api/users, /api/orders
   ❌ /api/getUser, /api/orderList

2. 请求方法对应操作
   GET    → 查询
   POST   → 新增
   PUT    → 全量更新
   PATCH  → 部分更新
   DELETE → 删除

3. 响应统一格式
   { "code": 200, "message": "success", "data": {...}, "timestamp": 1234567890 }

4. 分页接口传 page+size，返回 total+list
   ❌ pageSize/pageNum/offset/limit 各家一套

5. 时间字段统一格式
   ✅ yyyy-MM-dd HH:mm:ss
   ✅ 统一用 UTC 或东八区，文档注明

6. 枚举值文档化
   // 在 @Schema 的 description 里写清楚
   @Schema(description = "状态: 0-待支付, 1-已支付, 2-已取消")
   private Integer status;
```

## 修改默认的接口返回（重要）

默认有接口会暴露一些不需要的端点，生产环境可能成安全隐患。

```yaml
springdoc:
  paths-to-exclude:
    - /actuator/**
    - /error
```

或者通过分组隔离：

```java
@Bean
public GroupedOpenApi publicApi() {
    return GroupedOpenApi.builder()
            .group("对外接口")
            .pathsToMatch("/api/**")
            .build();
}

@Bean
public GroupedOpenApi adminApi() {
    return GroupedOpenApi.builder()
            .group("管理接口")
            .pathsToMatch("/admin/**")
            .build();
}
```

## 文档不写注释的后果

```java
@Data
public class OrderCreateReq {
    private String type;         // 我写的时候知道这是什么，一个月后：？
    private String source;
    private String ext;
}
```

**经验：** 不写 `@Schema` 注解的字段，前端看不到说明，只能来问你。一次两次之后他们就不再信任你的接口，转而自己去扒代码——然后误解你的逻辑，出 bug 了来吵。

## 踩坑记录

### 1. 生产环境暴露了 Swagger 文档

**现象：** 上线后发现 `https://prod.api/doc.html` 能访问，接口信息全部暴露

**原因：** 忘了关闭 Swagger

**解决：**
```yaml
# 多环境配置
# application-dev.yml
knife4j:
  enable: true

# application-prod.yml
springdoc:
  api-docs:
    enabled: false
  swagger-ui:
    enabled: false
```

### 2. 接口文档和实际接口不一致

**原因：** 改了参数忘了改注解

**原则：** 代码即文档，改了代码就要改注解。Code Review 时把注解变更作为必查项。
