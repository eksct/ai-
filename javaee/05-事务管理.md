# 事务管理

## 概述

事务是保证数据库操作原子性的核心机制。Spring 声明式事务（`@Transactional`）虽然方便，但很多人用错了都不知道——**生产事故大多和事务有关**。

## 设计原理

### Spring @Transactional 工作机制

```
@Transactional
    ↓
AOP 动态代理
    ↓
try {
    connection.setAutoCommit(false)
    你的业务代码 method()
    connection.commit()
} catch (Exception e) {
    connection.rollback()
    throw e
}
```

**关键点：** 依赖 AOP 代理。如果 AOP 没生效，事务也就没生效。

## 事务传播行为

```java
@Transactional(propagation = Propagation.REQUIRED)    // 默认：有事务就用，没有就新建
@Transactional(propagation = Propagation.REQUIRES_NEW) // 新建事务，挂起当前事务
@Transactional(propagation = Propagation.NESTED)       // 嵌套事务（JDBC savepoint）
@Transactional(propagation = Propagation.MANDATORY)    // 必须在事务中，否则抛异常
@Transactional(propagation = Propagation.NEVER)        // 不能在事务中
@Transactional(propagation = Propagation.SUPPORTS)     // 有就用，没有就算了
@Transactional(propagation = Propagation.NOT_SUPPORTED)// 不支持事务，挂起当前事务
```

### 生产常用组合

```java
// 独立事务，失败不影响主事务
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logOperation(Log log) {
    logMapper.insert(log);
}

// 主事务
@Transactional(rollbackFor = Exception.class)
public void processOrder(Order order) {
    orderMapper.insert(order);
    // 库存扣减失败不影响主事务？太天真了...
    // 如果用 REQUIRED，库存异常 → 主事务一起回滚
    // 所以需要上面的 REQUIRES_NEW 把日志单独拿出来
    logOperation(new Log("创建订单", order.getId()));
}
```

## 事务失效的 8 种场景（生产必知）

### 1. 同类中方法互相调用

```java
@Service
public class OrderService {
    public void createOrder(Order order) {
        // ...
        this.updateStock(order);  // @Transactional 不生效！！
    }

    @Transactional
    public void updateStock(Order order) {
        stockMapper.deduct(order.getProductId(), order.getQuantity());
    }
}
```

**原因：** `this.updateStock()` 直接调用了本对象方法，没有经过 AOP 代理

**解决：**
```java
// 方案一：把 updateStock 抽到另一个 Service
// 方案二：注入自己
@Service
public class OrderService {
    @Autowired
    private OrderService self;  // 注入代理对象

    public void createOrder(Order order) {
        self.updateStock(order); // 经过代理，事务生效
    }
}
```

### 2. 异常类型不对

```java
@Transactional
public void process() {
    try {
        // ...
    } catch (Exception e) {
        // 什么都没做，异常被吞了——事务不会回滚
    }
}

@Transactional
public void process2() {
    throw new RuntimeException();    // ✅ 默认回滚
    throw new BusinessException();   // ❌ BusinessException extends Exception
                                     // 默认不回滚！！！
}
```

**原因：** Spring 默认只回滚 `RuntimeException` 和 `Error`

**解决：**
```java
@Transactional(rollbackFor = Exception.class)  // 所有异常都回滚
public void process() {
    // 生产统一这么写，不要省
}
```

### 3. private 方法

```java
@Component
public class AccountService {
    @Transactional
    private void transfer() { }  // ❌ 不生效，AOP 不能代理 private 方法
}
```

### 4. final 方法

```java
@Transactional
public final void process() { }  // ❌ 不生效
```

### 5. 不同的线程

```java
@Transactional
public void process() {
    new Thread(() -> {
        // 新线程拿不到原事务的连接，事务不生效
    }).start();
}
```

### 6. 数据库引擎不支持

MySQL MyISAM 引擎不支持事务。**生产必须用 InnoDB。**

### 7. try-catch 吞掉异常

```java
@Transactional(rollbackFor = Exception.class)
public void process() {
    try {
        orderMapper.insert(order);
        throw new RuntimeException("失败");
    } catch (Exception e) {
        log.error("错误", e);
        // 注意：没抛出异常！事务不会回滚！
    }
}
```

### 8. 使用 @Transactional 加在接口上

```java
public interface OrderService {
    @Transactional  // ❌ 不推荐，接口上的 @Transactional 可能不生效
    void create(Order order);
}
```

**官方建议：** 加在实现类上，而不是接口上

## 事务隔离级别

```java
@Transactional(isolation = Isolation.DEFAULT)          // 默认（数据库的默认级别）
@Transactional(isolation = Isolation.READ_UNCOMMITTED) // 读未提交（脏读）
@Transactional(isolation = Isolation.READ_COMMITTED)   // 读已提交（Oracle 默认）
@Transactional(isolation = Isolation.REPEATABLE_READ)  // 可重复读（MySQL 默认）
@Transactional(isolation = Isolation.SERIALIZABLE)     // 串行化（性能极差）
```

**生产实践中：**
- **MySQL + SpringBoot 默认流程**：`REPEATABLE_READ`，够用
- **高并发扣库存场景**：序列化不行就用悲观锁/乐观锁
- **做数据统计/报表**：`READ_UNCOMMITTED` 可以接受脏读，提高性能

## 分布式事务

微服务场景下，`@Transactional` 管不了跨服务的事务。

### 解决方案选型

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 最终消息表 | 最终一致 | 高 | 低 | 通用 |
| Seata AT | 强一致 | 中 | 中 | 对一致性要求高 |
| TCC | 强一致 | 低 | 高 | 关键资金链路 |
| Saga | 最终一致 | 高 | 高 | 长事务 |

**中小团队推荐：** 用最终消息表（本地消息表 + 定时任务补偿），别一上来就上 Seata。

### 本地消息表示例

```java
@Transactional(rollbackFor = Exception.class)
public void createOrder(Order order) {
    // 1. 插入订单（主事务）
    orderMapper.insert(order);

    // 2. 插入消息记录（同一个本地事务！）
    EventMessage msg = new EventMessage();
    msg.setBizId(order.getId().toString());
    msg.setEventType("ORDER_CREATED");
    msg.setStatus(0); // 未发送
    msg.setContent(JSON.toJSONString(order));
    eventMessageMapper.insert(msg);

    // 3. 发送 MQ（如果这一步失败，事务回滚，消息也不会发出去）
    //    但这里已经出了事务，需要额外处理——见踩坑记录
}
```

## 踩坑记录

### 1. 大事务

**现象：** 一个 `@Transactional` 里处理了 10000 条数据，耗时 30 秒，数据库连接长时间不释放，连接池被打满。

**解决：**
```java
// ❌ 大事务
@Transactional
public void batchProcess(List<Order> orders) {
    for (Order order : orders) {
        processOrder(order);   // 一条一条处理，全在一个事务里
    }
}

// ✅ 拆分
public void batchProcess(List<Order> orders) {
    for (Order order : orders) {
        processOrder(order);   // 每条独立事务
    }
}
```

### 2. 事务中调用远程 RPC

```java
@Transactional(rollbackFor = Exception.class)
public void createOrder(Order order) {
    orderMapper.insert(order);
    httpClient.doSomething();  // 外部调用！事务还在持续
    // 如果外部调用超时 30 秒，事务就一直持有连接
}
```

**原则：** 不在事务中做远程调用。把远程调用提到事务之后。

### 3. @Transactional 加在 Controller 上

不推荐。Service 层才是事务的合理边界。Controller 加事务等于把所有 HTTP 请求都包在事务里，连接池很快就满了。
