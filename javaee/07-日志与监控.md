# 日志与监控

## 概述

日志不是写给自己看到，而是写给**未来的排查者**。生产环境没有 debug 给你打断点，日志是你唯一的线索。

好的日志规范 = 出问题时能快速定位，而不是在几万行日志里 grep。

## 日志框架选型

| 组合 | 场景 |
|------|------|
| Logback | Spring Boot 默认，够用 |
| Log4j2 | 性能更好（异步日志），配置复杂 |
| 日志门面 | SLF4J（永远用这个，不要直接用 Logback API）|

**原则：** 代码里只引 `org.slf4j.Logger`，不直接依赖具体实现

## 日志配置

```xml
<!-- logback-spring.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<configuration scan="true" scanPeriod="60 seconds">

    <!-- 从 application.yml 读取 -->
    <springProperty scope="context" name="APP_NAME" source="spring.application.name"/>
    <springProperty scope="context" name="LOG_PATH" source="logging.file.path" defaultValue="/data/logs"/>

    <!-- 控制台输出 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="ch.qos.logback.classic.encoder.PatternLayoutEncoder">
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 文件输出：按天滚动，保留 30 天 -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>${LOG_PATH}/${APP_NAME}.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>${LOG_PATH}/${APP_NAME}.%d{yyyy-MM-dd}.log.gz</fileNamePattern>
            <maxHistory>30</maxHistory>
            <totalSizeCap>10GB</totalSizeCap>
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 错误日志单独输出 -->
    <appender name="ERROR_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>${LOG_PATH}/${APP_NAME}-error.log</file>
        <filter class="ch.qos.logback.classic.filter.ThresholdFilter">
            <level>ERROR</level>
        </filter>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>${LOG_PATH}/${APP_NAME}-error.%d{yyyy-MM-dd}.log.gz</fileNamePattern>
            <maxHistory>60</maxHistory>
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 异步输出（生产重要） -->
    <appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
        <queueSize>1024</queueSize>
        <discardingThreshold>0</discardingThreshold>
        <neverBlock>true</neverBlock>
        <appender-ref ref="FILE"/>
    </appender>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="ASYNC_FILE"/>
        <appender-ref ref="ERROR_FILE"/>
    </root>

</configuration>
```

## 日志怎么打（生产经验）

```java
// ❌ 不要这样打日志
log.info("创建订单");
log.info("订单创建成功，订单号：" + orderNo + "，金额：" + amount);

// ✅ 用占位符，不用字符串拼接
log.info("创建订单成功，orderNo={}, amount={}", orderNo, amount);
```

### 异常日志规范

```java
// ❌ 不推荐
try {
    // ...
} catch (Exception e) {
    log.error("出错了");                       // 没打异常栈
    log.error("出错了: " + e.getMessage());    // 没打异常堆栈
}

// ✅ 推荐：把异常对象作为最后一个参数
try {
    // ...
} catch (Exception e) {
    log.error("处理订单失败, orderNo={}", orderNo, e);  // 打印完整堆栈
}
```

### 不同级别的使用场景

| 级别 | 使用场景 |
|------|----------|
| ERROR | 影响业务流程的异常，需要人工处理 |
| WARN | 不影响流程但需要注意的情况（如重试、限流） |
| INFO | 业务流程的关键节点（创建订单、支付回调）|
| DEBUG | 开发调试，**生产关闭** |
| TRACE | 极少使用 |

## 日志链路追踪（MDC）

```java
// 过滤器：在每个请求入口设置 traceId
@Component
public class TraceIdFilter implements Filter {
    private static final String TRACE_ID = "traceId";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        try {
            String traceId = ((HttpServletRequest) request).getHeader("X-Trace-Id");
            if (StringUtils.isBlank(traceId)) {
                traceId = UUID.randomUUID().toString().replace("-", "").substring(0, 16);
            }
            MDC.put(TRACE_ID, traceId);
            chain.doFilter(request, response);
        } finally {
            MDC.remove(TRACE_ID);  // 必须清理
        }
    }
}

// logback 配置中打印 traceId
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level [%X{traceId}] %logger{36} - %msg%n</pattern>
```

**为什么要链路追踪？** 没有 traceId，你没法把"用户 A 的请求"分散到各个服务、各个线程的日志串起来。

```yaml
management:
  endpoints:

      exposure:
        include: health,info,metrics,env,prometheus
      base-path: /actuator
  endpoint:
    health:
      show-details: always

      enabled: false         # 生产环境建议关掉 env，避免泄露配置
```

## 踩坑记录

### 1. 生产日志把磁盘打满

**现象：** 某天服务挂了，排查发现磁盘 100%，日志文件几百 GB

**原因：** 某次紧急上线把 `log.info("接收订单: " + JSON.toJSONString(order))` 打在了大促高频接口里

**教训：**
- 高频接口的日志用 WARN 或 DEBUG
- INFO 级别的日志要有节制
- 不要等到出事才想起来——日志监控应该是日常

### 2. 日志泄露敏感信息

**现象：** 日志里打印了用户手机号和身份证号，运维查看日志时隐私泄露

**解决：** 脱敏工具类
```java
public class LogUtils {
    public static String maskPhone(String phone) {
        if (StringUtils.isBlank(phone)) return "";
        return phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
    }

    public static String maskIdCard(String idCard) {
        if (StringUtils.isBlank(idCard)) return "";
        return idCard.replaceAll("(\\d{4})\\d{10}(\\w{4})", "$1**********$2");
    }
}

// 日志中调用
log.info("用户手机号: {}", LogUtils.maskPhone(user.getPhone()));
```

### 3. AsyncAppender 丢日志

**现象：** 服务非正常关闭（kill -9），最后几条日志丢失

**原因：** 异步日志有缓冲队列，没来得及写入

**权衡：** 这是异步日志的代价，生产环境可以接受。如果你的业务场景**一条都不能丢**，用同步输出。
