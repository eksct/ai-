# 生产选型与坑点

## 概述

这篇不讲基础，只讲**实际生产里的决策和教训**。低代码转 Java 最容易踩的坑就是——以为"跑起来了就是会了"。

## 线程池配置（生产必调）

### 错误的做法

```java
// ❌ 直接抄网上，不考虑业务场景
Executors.newFixedThreadPool(10);
// 无界队列，峰值请求堆积到 OOM

Executors.newCachedThreadPool();
// 无限创建线程，CPU/内存打满

// ❌ 或者干脆不配，用 Tomcat 默认 200 线程
// 200 个线程同时处理数据库操作，连接池撑爆
```

### 正确的做法

```java
@Configuration
public class ThreadPoolConfig {
    @Bean("bizExecutor")
    public ThreadPoolExecutor bizExecutor() {
        // 根据业务的 IO 密集程度计算
        // IO 密集型：核心线程数 = CPU核数 * 2
        // CPU 密集型：核心线程数 = CPU核数 + 1
        int cores = Runtime.getRuntime().availableProcessors();

        return new ThreadPoolExecutor(
            cores * 2,                      // corePoolSize
            cores * 4,                      // maxPoolSize
            60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1024), // 有界队列！
            new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
        );
    }
}
```

**更重要的——监控线程池：**
```java
// 生产必须加监控
@Component
public class ThreadPoolMonitor {
    @Scheduled(fixedRate = 60000)
    public void monitor() {
        ThreadPoolExecutor executor = ...;
        log.info("线程池状态: active={}, queued={}, completed={}, poolSize={}",
            executor.getActiveCount(),
            executor.getQueue().size(),
            executor.getCompletedTaskCount(),
            executor.getPoolSize()
        );

        // 队列积压超过阈值告警
        if (executor.getQueue().size() > 500) {
            alertService.sendAlert("线程池队列积压", executor.getQueue().size());
        }
    }
}
```

## 慢 SQL 治理

### 发现的时机

不是等用户投诉才处理，而是主动发现。

```yaml
# 开启慢 SQL 日志（MySQL）
slow_query_log = ON
long_query_time = 1  # 超过 1 秒就算慢
```

```java
// MyBatis-Plus 拦截器统计执行时间
@Component
@Intercepts({@Signature(type = Executor.class, method = "update", args = {MappedStatement.class, Object.class}),
             @Signature(type = Executor.class, method = "query", args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})})
public class SqlPerformanceInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return invocation.proceed();
        } finally {
            long cost = System.currentTimeMillis() - start;
            if (cost > 1000) {  // 超过 1 秒告警
                log.warn("慢 SQL ({}ms): {}", cost, getSql(invocation));
                // 发到告警平台
            }
        }
    }
}
```

### 常见慢 SQL 原因

```sql
-- 1. 没加索引（最典型）
SELECT * FROM `order` WHERE user_id = 123;     -- ❌ user_id 没索引

-- 2. 索引失效
SELECT * FROM `order` WHERE DATE(create_time) = '2024-01-01';  -- ❌ 函数导致索引失效
SELECT * FROM `order` WHERE create_time >= '2024-01-01' AND create_time < '2024-01-02';  -- ✅

-- 3. 大分页
SELECT * FROM `order` LIMIT 100000, 20;  -- ❌ 越往后越慢

-- 4. 隐式类型转换
SELECT * FROM `user` WHERE phone = 13800138000;  -- ❌ phone 是 varchar
SELECT * FROM `user` WHERE phone = '13800138000'; -- ✅

-- 5. 联表过多
SELECT ... FROM a JOIN b JOIN c JOIN d JOIN e WHERE ...  -- 5 张表以上就要审视了
```

## 接口性能优化

### 缓存策略

```
查询类接口：先查缓存 → 命中直接返回 → 未命中查库 → 回填缓存
写操作：    先写库 → 删除/更新缓存（不用先更新缓存）
```

**缓存更新的坑：**
- 先更新缓存再写库 → 写库失败，缓存写脏
- 先删缓存再写库 → 并发读又写回旧数据
- **推荐：先写库再删缓存（Cache Aside Pattern）**

```java
public ApiResult<Void> batchProcess(@RequestBody List<Long> ids) {
```

```java

    orderMapper.insert(order);  // N 次网络往返

// ✅ 批量插入
```

## 代码层面的坑

### 1. BigDecimal 初始化

```java
// ❌ double 构造存在精度问题
BigDecimal d = new BigDecimal(0.1);   // 0.100000000000000005551115123...

// ✅ 字符串构造
BigDecimal d = new BigDecimal("0.1");

// ✅ 或使用 valueOf（推荐）
BigDecimal d = BigDecimal.valueOf(0.1);
```

### 2. 日期格式化线程安全问题

```java
// ❌ SimpleDateFormat 线程不安全
private static final SimpleDateFormat SDF = new SimpleDateFormat("yyyy-MM-dd");

// ✅ 用 DateTimeFormatter（Java 8+，不可变、线程安全）
private static final DateTimeFormatter DTF = DateTimeFormatter.ofPattern("yyyy-MM-dd");
```

### 3. equals 和 ==

```java
// ❌ Long 比较
Long id1 = 128L;
Long id2 = 128L;
id1 == id2          // false（Long 缓存 -128~127）
id1.equals(id2)     // true

// ✅ 所有对象比较都用 equals
Objects.equals(id1, id2);

// ❌ 枚举比较
// 可以用 ==，枚举的 == 和 equals 等价
```

### 4. try-with-resources

```java
// ❌ 手动关闭容易漏
InputStream is = null;
try {
    is = new FileInputStream("file");
    // ...
} finally {
    if (is != null) {
        try { is.close(); } catch (IOException e) {}
    }
}

// ✅ try-with-resources（自动关闭）
try (InputStream is = new FileInputStream("file")) {
    // ...
}
```

## 架构级的坑

### 1. 单体应用过度设计

**现象：** 一个订单管理项目，上来就拆了 8 个微服务，结果每个服务只有几张表，接口调用链变长，排查问题需要查 5 个服务日志

**原则：** 
- 团队 < 20 人 → 单体优先，模块化拆分即可
- 业务复杂到单体内无法维护 → 考虑拆微服务
- 不要为了微服务而微服务

### 2. 大事务 + 远程调用

```java
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    // 调用外部支付网关（可能 5 秒超时）
    paymentService.pay(order);  // 事务不释放，连接池打满
}
```

**原则：** 事务里不做远程调用。如果一定要，把远程调用移到事务外。

### 3. 日志打太多

**现象：** 异步日志都写不过来，业务线程卡在日志上

**解决：** 高频接口用 WARN，日志异步队列监控告警

### 4. 缺少限流

**现象：** 大促时某个接口被刷了 1000 QPS，数据库被打死

**解决：**
```java
// 最简单的限流：Sentinel / 或基于 Redis 的滑动窗口
@Component
public class RateLimiter {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    public boolean tryAcquire(String key, int limit, int windowSeconds) {
        // 滑动窗口限流
        String script = """
            local key = KEYS[1]
            local now = tonumber(ARGV[1])
            local window = tonumber(ARGV[2])
            local limit = tonumber(ARGV[3])
            redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
            local count = redis.call('ZCARD', key)
            if count < limit then
                redis.call('ZADD', key, now, now)
                return 1
            end
            return 0
        """;
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Arrays.asList(key),
            System.currentTimeMillis(), windowSeconds * 1000, limit
        );
        return Long.valueOf(1).equals(result);
    }
}
```

## 总结（给低代码转 Java 的同学）

1. **先跑起来，再深入**：快速搭一个 CRUD 项目跑通全流程
2. **不要重复造轮子**：能用框架的用框架，Spring Boot 社区很成熟
3. **日志是你唯一的 debug 工具**：没有 IDE 断点，靠日志
4. **每行代码都有性能成本**：循环里调个 Redis 可能把服务打垮
5. **"能跑"和"生产可用"之间差 10 倍**：考虑异常、限流、监控、降级
