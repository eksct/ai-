# 正向代理与反向代理

## 概述

代理的本质是中间人。正向代理代理客户端，反向代理代理服务端。

## 正向代理（Forward Proxy）

```
客户端 → 代理服务器 → 目标服务器
（知道目标地址，但无法直连，让代理帮我去）
```

场景：翻墙、公司内网访问外网、爬虫换 IP。

特点：
- 客户端**主动配置**代理地址（浏览器设置、PAC 文件、VPN）
- 目标服务器不知道真正的客户端是谁（看到的是代理的 IP）
- 代理的是"我"（客户端）

## 反向代理（Reverse Proxy）

```
客户端 → 反向代理 → 真实服务器
（只知道代理地址，不知道目标在哪，代理帮我找）
```

场景：Nginx 挡在前端后面、负载均衡、统一 HTTPS 入口。

特点：
- 客户端**无感知**，不需要任何配置
- 客户端只知道 Nginx 的地址，真实服务器完全不可见
- 代理的是"他们"（服务端）

## 对比

| | 正向代理 | 反向代理 |
|---|---|---|
| 代理谁 | 客户端 | 服务端 |
| 谁配置 | 客户端主动配置 | 服务端部署，客户端不知情 |
| 客户端知道目标吗 | 知道 | 不知道 |
| 服务端知道客户端吗 | 不知道（代理 IP） | 知道（通过 X-Forwarded-For） |
| 典型工具 | Shadowsocks、V2Ray、Squid | Nginx、HAProxy、Traefik |

## Nginx 做反向代理的本质

```
用户浏览器 → Nginx（门面）
               ├ → Java 应用（内网 192.168.1.10:8080）
               ├ → Python 服务（内网 192.168.1.11:5000）
               └ → 静态文件（本地磁盘）
```

浏览器只跟 Nginx 打交道，背后有多少台服务器、在哪、跑的什么语言——对用户完全透明。这就是反向代理的核心价值。
