# Linux 进程与系统管理

## 概述

线上出问题第一件事：看进程是不是活着？内存够不够？磁盘满没满？CPU 谁在吃？这篇的命令就是**查问题三板斧**。

## 进程管理

### ps — 查看进程快照

```bash
ps aux                          # 所有进程（完整列表）
ps auxf                         # 树形结构（父子进程关系）
ps aux | grep nginx             # 找特定进程
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%mem | head 10
# ↑ 按内存排序，取前10
```

输出解读：`USER PID %CPU %MEM VSZ RSS TTY STAT START TIME COMMAND`

### top / htop — 实时监控

```bash
top                             # 系统实时状态
# 快捷键：
# P    按 CPU 排序
# M    按内存排序
# 1    看每个 CPU 核心
# q    退出

htop                            # 更好看的 top（需要安装）
```

`top` 输出关键行：

```
load average: 0.50, 0.30, 0.20     # 1/5/15 分钟平均负载
                                     # < CPU 核心数 就没问题
%Cpu(s): 2.5 us, 1.2 sy, 0.0 ni, 95.3 id, 0.5 wa, 0.5 st
# us=用户进程 sy=系统 ni=优先级 id=空闲 wa=IO等待 st=被虚拟机抢走
```

### kill — 终止进程

```bash
kill PID                        # 优雅终止（发 SIGTERM）
kill -9 PID                     # 强制杀死（发 SIGKILL，进程无法捕获）
kill -15 PID                    # 同 kill（SIGTERM）
kill -19 PID                    # 暂停进程（SIGSTOP）
kill -18 PID                    # 恢复进程（SIGCONT）
pkill nginx                     # 按名字杀
pkill -f "java -jar"            # 按命令行匹配杀
killall nginx                   # 杀掉所有同名进程
```

**`kill -9` 是最后手段**：进程可能正在写文件，直接 -9 会丢数据。先 `kill`（SIGTERM），等几秒不行再 `-9`。

### nohup — 后台运行

```bash
nohup java -jar app.jar > app.log 2>&1 &
# ↑                 ↑日志    ↑错误也到日志 ↑后台运行

# 更现代的方案（推荐）：
systemctl start myapp           # 作为服务管理
docker run -d myapp             # 容器化运行
```

## 系统资源

### free — 内存查看

```bash
free -h
#               total   used   free   shared   buff/cache   available
# Mem:           15G    6.2G   5.1G    1.2G        3.7G       8.5G
```

关注 **available** 列——它是真正可用的内存（包含可回收的缓存）。used 大不代表内存不够。

### df — 磁盘查看

```bash
df -h                           # 所有分区使用情况
df -h /data                     # 只看 /data 分区
df -i                           # inode 使用情况（小文件太多也会满）
```

### du — 目录大小

```bash
du -sh /var/log                 # 总大小
du -sh * | sort -hr             # 当前目录下各子目录大小排序
du -h --max-depth=1 /data       # 只展开一层
```

### 找磁盘大文件

```bash
find / -type f -size +100M -exec ls -lh {} \; | sort -k5 -hr | head 10
```

## 服务管理

### systemctl — systemd 服务

```bash
systemctl start nginx           # 启动
systemctl stop nginx            # 停止
systemctl restart nginx         # 重启
systemctl reload nginx          # 重载配置（不中断服务）
systemctl status nginx          # 查看状态

systemctl enable nginx          # 开机自启
systemctl disable nginx         # 取消开机自启
systemctl is-enabled nginx      # 查看是否开机自启
```

### journalctl — 查看系统日志

```bash
journalctl -u nginx             # 看 nginx 的日志
journalctl -u nginx --since "1 hour ago"
journalctl -u nginx -f          # 实时跟踪
journalctl --disk-usage         # 日志占用多少磁盘
```

### 常见日志文件

```bash
/var/log/syslog          # 系统总日志（Ubuntu/Debian）
/var/log/messages        # 系统总日志（CentOS/RHEL）
/var/log/nginx/          # Nginx 日志
/var/log/mysql/          # MySQL 日志
/var/log/dmesg           # 内核日志
```

## 生产实践

### 排查流程

线上出问题，按顺序查：

```bash
# 1. 系统负载
top                        # CPU 和内存

# 2. 磁盘
df -h                      # 磁盘满没满
du -sh /var/log            # 日志有没有爆

# 3. 内存
free -h                    # 内存够不够

# 4. 具体服务
systemctl status nginx     # 服务活着没
journalctl -u nginx -n 50  # 最近日志

# 5. 日志
tail -f /var/log/nginx/access.log
```

### 常用组合

```bash
# 查哪个进程吃内存最多
ps aux --sort=-%mem | head 5

# 查哪个进程吃 CPU 最多
ps aux --sort=-%cpu | head 5

# 清缓存（仅限测试环境）
echo 3 > /proc/sys/vm/drop_caches

# 限制日志大小（doker 容器）
find /var/lib/docker/containers -name "*.log" -delete
```

### 踩坑记录

1. **磁盘满的隐藏原因**：`df -h` 显示还有空间，但应用写不了文件——一般是 **inode 满了**（小文件太多）。`df -i` 检查
2. **`kill -9` 酿成大祸**：线上直接 `kill -9` MySQL，结果 MySQL 启动时 crash recovery 跑了半小时。先 `kill`（SIGTERM）让进程自己善后
3. **`nohup` 后台进程失控**：`nohup ... &` 跑起来的进程没了日志输出，不知道它在干嘛。解决方案：用 `tmux` 或 `screen` 替代——可以随时 reconnect 看输出
4. **`journalctl` 日志不清理**：`systemd` 日志默认无限制增长，几年不清理可能占几十 GB。设 `SystemMaxUse=500M` 限制日志上限

## 总结

**查问题三部曲：** `top` 看负载 → `df -h` 看磁盘 → `tail -f` 看日志。`kill -9` 是核弹，能用常规 `kill` 就不用它。
