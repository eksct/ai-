# Redis 主从复制与哨兵

## 概述

单个 Redis 实例挂了服务就断。**主从复制**解决数据冗余（多存几份），**哨兵**解决自动故障转移（主挂了自动选新主）。两个配合实现 Redis 高可用。

## 设计原理

### 为什么需要主从

单机 Redis 的风险：

- 机器挂了 → 所有请求打穿到 DB → DB 跟着挂
- 重启恢复期间 → 服务不可用
- 读压力大 → 单机抗不住

主从架构把写请求隔离到主库，读请求分散到从库，同时从库作为主库的热备份。

### 一主多从架构

```
        客户端
          |
     +----+----+
     |         |
   主库       哨兵（Sentinel）
     |         |
  +--+--+      |
  |     |      |
从库A  从库B

主库：可读可写
从库：只读，同步主库数据，主挂了被选为新主
哨兵：监控主从健康，自动故障转移
```

## 主从复制

### 复制原理（全量 + 增量）

```
第一次同步（全量复制）：
1. 从库向主库发送 PSYNC
2. 主库 BGSAVE 生成 RDB → 发送给从库
3. 主库把 RDB 生成期间的增量命令写入缓冲区
4. RDB 传完 → 从库加载 → 再发送缓冲区增量命令

后续同步（增量复制）：
主库把写命令实时推送给从库（基于长连接）
```

全量复制只在从库第一次连主库或断连太久时发生。之后都是增量的，开销很小。

### 配置

#### 命令行配置
```bash
# 从库上执行：指定主库地址
SLAVEOF 192.168.1.10 6379
# 或新版命令
REPLICAOF 192.168.1.10 6379
```

#### 配置文件配置
```conf
# 从库配置文件
replicaof 192.168.1.10 6379
replica-read-only yes
```

### 验证

```bash
# 主库查看从库列表
INFO replication

# 从库查看复制状态
INFO replication
ROLE
```

## 哨兵（Sentinel）

### 是什么

一个独立的分布式监控系统，监控所有 Redis 节点。发现主库挂了，自动选一个从库升级为新主，修改其他从库指向新主。

### 架构

哨兵通常是 3 个或 5 个节点（奇数），部署在不同机器上。**哨兵之间互相通信**，用投票决定主库是否真的挂了。

### 故障转移流程

```
1. 哨兵 A 发现主库 ping 超时 → 标记"主观下线"
2. 哨兵 A 通知其他哨兵确认
3. 多数哨兵（> 一半）都认为主库挂了 → 标记"客观下线"
4. 哨兵们选出一个 Leader
5. Leader 从从库中选一个作为新主（优先级最高、数据最新、runid 最小）
6. 其他从库切换为新主的从库
7. 旧主恢复了 → 变成新主的从库
```

### 配置 sentinel.conf

```conf
# 监控 master 实例，2 个哨兵确认即为客观下线
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 5000     # 5 秒没响应即下线
sentinel failover-timeout mymaster 60000           # 故障转移超时
sentinel parallel-syncs mymaster 1                 # 同时几个从库同步新主
```

启动：`redis-sentinel sentinel.conf`

### 哨兵 vs 直接写死 IP

| 方案 | 主库切换后 | 适用场景 |
|------|-----------|----------|
| 客户端直连主库 IP | IP 变了，客户端得重启改配置 | 小项目，能接受 downtime |
| 哨兵 + 客户端 SDK | 自动获取新主 IP | 生产环境，追求高可用 |

Java（Spring Data Redis）和 Go（go-redis）都有哨兵模式的内置支持：

```go
// Go: go-redis 哨兵模式
rdb := redis.NewFailoverClient(&redis.FailoverOptions{
    MasterName:    "mymaster",
    SentinelAddrs: []string{"192.168.1.1:26379", "192.168.1.2:26379"},
})
```

```java
// Spring Data Redis 哨兵模式
@Bean
public RedisConnectionFactory redisConnectionFactory() {
    RedisSentinelConfiguration sentinelConfig = new RedisSentinelConfiguration()
        .master("mymaster")
        .sentinel("192.168.1.1", 26379)
        .sentinel("192.168.1.2", 26379);
    return new JedisConnectionFactory(sentinelConfig);
}
```

## 生产实践

### 常用架构

| 规模 | 架构 | 说明 |
|------|------|------|
| < 10GB | 主 + 从 + 3 哨兵 | 一主一从防丢数据 |
| 10~50GB | 主 + 2 从 + 3 哨兵 | 读写分离，读分摊到从库 |
| > 50GB | Redis Cluster | 数据分片，天然高可用 |

### 踩坑记录

1. **从库默认不顶住读流量**：如果只是为高可用加从库不开读分离，从库资源浪费。读多场景一定要改客户端配置走从库读
2. **哨兵脑裂（Split-Brain）**：网络分区导致两边都认为主库挂了，各选一个新主。网络恢复后数据不一致。解决方案：`sentinel monitor` 的 quorum 值设到多数（>= n/2 + 1）
3. **全量复制时主库压力大**：主库 BGSAVE 生成 RDB 会 fork，如果同时有多个从库启动全量复制，主库 CPU 和磁盘 IO 飙升。控制 `sentinel parallel-syncs 1`，逐个同步
4. **故障转移期间有写入失败**：从哨兵判定主库下线到新主上线，有几十秒的窗口。写请求会失败。业务层要加重试

## 总结

主从复制解决数据冗余，哨兵解决自动切换。两者配合是 Redis 生产环境的**最低门槛高可用方案**。但哨兵没有负载均衡和数据分片能力，再往上就是 Redis Cluster。
