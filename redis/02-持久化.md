# Redis 持久化

## 概述

Redis 数据在内存中，进程退出数据就丢。持久化就是把内存数据**写入磁盘**，重启后恢复。Redis 提供两种方案：**RDB**（快照）和 **AOF**（追加日志），可单独用也可组合。

## 设计原理

持久化解决的核心问题只有一个：**Redis 重启了，数据还在不在？**

- 缓存场景：丢了无所谓，从 DB 重新加载就行 → **不需要持久化**
- 数据存储场景：丢了就没了 → **需要持久化**

## RDB（Redis Database）

### 是什么

按指定间隔把内存数据**全量 dump** 到磁盘，生成一个紧凑的二进制文件 `.rdb`。

### 触发方式

```bash
# 手动触发
SAVE                    # 同步阻塞，生产别用
BGSAVE                  # 异步 fork 子进程写入，推荐

# 自动触发（配置 save 指令，满足任一即触发）
save 900 1              # 900 秒内至少 1 次写操作
save 300 10             # 300 秒内至少 10 次写操作
save 60 10000           # 60 秒内至少 10000 次写操作
```

### 工作原理（Copy-on-Write）

```
1. fork() 出子进程
2. 子进程将共享内存数据写入临时 RDB 文件
3. 写入完成，原子替换旧 RDB 文件
```

`fork` 用的是 COW（写时复制），子进程刚出来时共享父进程内存页，只有父进程继续写数据时才复制修改的页。这意味着：

- `BGSAVE` 期间即使有写入，RDB 也是**某一时刻的完整快照**
- fork 开销和内存大小相关，大实例（>10GB）fork 可能卡住毫秒到秒级

## AOF（Append-Only File）

### 是什么

记录**每一条写命令**到日志文件，重启时重新执行这些命令来恢复数据。类似 MySQL 的 binlog。

### 配置

```conf
appendonly yes
appendfsync everysec       # 刷盘策略
```

| appendfsync | 说明 | 可靠性 | 性能 |
|-------------|------|--------|------|
| always | 每条命令都写磁盘 | 最高，最多丢一条 | 最慢 |
| everysec | 每秒刷一次 | 最多丢 1 秒数据 | 推荐 |
| no | 交给操作系统决定 | 可能丢几十秒数据 | 最快 |

### AOF 重写（Rewrite）

AOF 文件无限增长，需要定期重写：把内存里的数据反向生成一批最小命令，替换旧文件。

```bash
BGREWRITEAOF              # 手动触发重写
```

自动触发配置：

```conf
auto-aof-rewrite-percentage 100    # 文件比上次大了一倍
auto-aof-rewrite-min-size 64mb     # 且至少 64MB
```

## RDB vs AOF

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 文件大小 | 小（二进制压缩） | 大（文本协议） |
| 恢复速度 | 快（直接加载） | 慢（逐条执行命令） |
| 数据安全性 | 丢上一次快照后的数据 | 最多丢 1 秒(everysec) |
| 对性能影响 | fork 子进程，COW 有内存开销 | 写日志有磁盘 IO |
| 人类可读 | 二进制，不可读 | 文本，可读可修改 |
| 适用场景 | 备份、快速恢复 | 高可靠性、数据防丢 |

### 官方推荐

Redis 配置文件里的默认策略：

```conf
# 默认：同时开启
save 900 1
save 300 10
save 60 10000
appendonly yes
appendfsync everysec
```

**RDB + AOF 同时开**：既享受 RDB 的快速恢复，又享受 AOF 的高可靠性。重启时 Redis 优先加载 AOF（因为数据更完整）。

## 生产实践

### 常用场景

| 场景 | 方案 | 理由 |
|------|------|------|
| 纯缓存，允许丢数据 | 关持久化 | 省磁盘 IO 和 fork 开销 |
| 数据存储，不能丢 | AOF + RDB 都开 | RDB 做备份，AOF 保实时 |
| 大数据量快速恢复 | RDB 为主 | AOF 恢复太慢 |
| 误操作紧急恢复 | AOF 文件 | 可以编辑 AOF 删掉误操作命令再重启 |

### 踩坑记录

1. **RDB fork 导致延迟抖动**：实例内存 > 10GB 时，fork 可能卡住几十毫秒到 1 秒。解决方案：用物理机大内存减少 fork 频率，或使用 Redis Cluster 拆分实例
2. **AOF 文件损坏**：机器宕机导致 AOF 尾部不完整。用 `redis-check-aof --fix` 修复，会截掉损坏部分，最多丢最后几条命令
3. **大 Key 导致 RDB 膨胀**：一个 Key 存了几百万元素的 Set，RDB 时这个 Key 会占大量空间。用 `DEBUG HTSTATS key` 或 `MEMORY USAGE key` 排查大 Key
4. **误以为开了 AOF 就万无一失**：AOF everysec 最多丢 1 秒数据。如果机器整机宕机、磁盘损坏，还是可能丢。重要数据除了 Redis 持久化，还要定期备份 RDB 到远程存储

## 总结

持久化不是银弹——先想清楚你的场景能不能接受丢数据。能接受就别开，省资源。不能接受就 RDB + AOF 都开，加定期备份。不要不配置就用默认值，理解 `save` 和 `appendfsync` 每个参数的含义。
