# Redis 概述与核心数据结构

## 概述

Redis（Remote Dictionary Server）是一个基于内存的 Key-Value 存储系统，通常用作缓存、消息队列和轻量级数据库。相比传统数据库，它的核心优势是**速度**——单机 QPS 可达 10 万+。

## 设计原理

### 为什么这么快

- **纯内存操作**：数据在内存中，读写是纳秒级
- **单线程模型**（v6.0 前）：避免锁竞争和上下文切换，所有操作原子性天然保证
- **IO 多路复用**：用 epoll 处理大量客户端连接，一个线程管所有
- **高效数据结构**：底层用压缩列表、跳表等优化存储

> v6.0 后网络 IO 改为多线程，但命令执行仍然是单线程。这点没变。

### 和 Memcached 比怎么选

| 特性 | Redis | Memcached |
|------|-------|-----------|
| 数据类型 | 字符串/列表/哈希/集合/有序集合 + 扩展类型 | 仅字符串 |
| 持久化 | RDB + AOF 支持 | 不支持 |
| 过期策略 | 6 种淘汰策略 + TTL | LRU 淘汰 |
| 集群方案 | 主从/哨兵/Cluster | 客户端分片 |
| 适用场景 | 缓存 + 业务数据 + 消息队列 | 纯缓存 |

选型建议：新项目无脑 Redis。Memcached 的优势（纯内存、极简）在 Redis 面前已经不明显。

## 核心数据结构和命令

### String（字符串）

最基础的类型，value 上限 512MB。底层是 SDS（Simple Dynamic String），相比 C 字符串预分配空间、O(1) 长度，避免缓冲区溢出。

```bash
SET name "redis"
GET name
SETEX token 3600 "abc123"    # 带过期时间
INCR counter                  # 原子自增（秒杀库存场景）
MSET k1 v1 k2 v2              # 批量操作减少 RTT
```

```go
// Go: go-redis
val, err := rdb.Get(ctx, "name").Result()
if err == redis.Nil {
    fmt.Println("key not found")
}
cnt, err := rdb.Incr(ctx, "counter").Result()
```

### List（列表）

基于双向链表（quicklist 结构），适合**消息队列**和**时间线**场景。

```bash
LPUSH queue task1              # 左侧入队
RPUSH queue task2              # 右侧入队
LPOP queue                     # 左侧出队
BLPOP queue 5                  # 阻塞弹出，超时5秒——实现简单队列
LRANGE queue 0 -1              # 遍历全部
```

```go
// Go: 生产环境用 Redis Stream 替代 List 做消息队列
len, err := rdb.LLen(ctx, "queue").Result()
```

**生产注意**：`LLEN` 是 O(1) 的，放心用。但 `LRANGE` 对大列表是 O(n)，慎用。

### Hash（哈希）

类似 Java 的 `Map<String, String>` 或 Go 的 `map[string]string`，适合存对象字段。

```bash
HSET user:1001 name "张三" age "30"
HGET user:1001 name
HGETALL user:1001              # 返回所有字段（小对象可以用）
HMSET user:1002 name "李四" age "25" city "北京"
```

```go
// Go
err := rdb.HSet(ctx, "user:1001", "name", "张三", "age", 30).Err()
name, err := rdb.HGet(ctx, "user:1001", "name").Result()
```

设计上应该用单 key 存一个对象，而不是 `user:1001:name` 和 `user:1001:age` 分开。用 Hash 省内存、方便管理。

### Set（集合）

无序、自动去重，适合**标签/关注/抽奖**场景。

```bash
SADD tags:article:42 "golang" "redis" "tutorial"    # 添加
SMEMBERS tags:article:42                            # 获取全部
SINTER tags:article:42 tags:article:43              # 交集（共同标签）
SISMEMBER tags:article:42 "golang"                  # 判断是否存在
```

```go
// Go
added, err := rdb.SAdd(ctx, "tags:article:42", "golang", "redis").Result()
members, err := rdb.SMembers(ctx, "tags:article:42").Result()
```

### ZSet（有序集合）

每个元素带一个 score，按 score 排序。底层用**跳表 + 哈希表**实现，是 Redis 最精妙的数据结构。

```bash
ZADD leaderboard 100 "player1" 85 "player2" 200 "player3"    # score: 分数
ZRANGE leaderboard 0 2 WITHSCORES                             # 前三名
ZRANK leaderboard "player1"                                   # 排名
ZINCRBY leaderboard 10 "player1"                              # 加分
```

```go
// Go
err := rdb.ZAdd(ctx, "leaderboard", 
    redis.Z{Score: 100, Member: "player1"},
    redis.Z{Score: 85, Member: "player2"},
).Err()
rank, err := rdb.ZRank(ctx, "leaderboard", "player1").Result()
```

### 选型速查

| 场景 | 推荐结构 | 原因 |
|------|----------|------|
| 缓存简单值 | String | 最直接 |
| 对象缓存 | Hash | 省内存，可单独读写字段 |
| 消息队列 | Stream（高可靠）/ List（简单） | Stream 支持消费者组 |
| 排行榜 | ZSet | 天然排序 |
| 去重统计 | Set / HyperLogLog | Set 精确，HLL 省内存 |
| 位图统计 | Bitmap / BitField | 亿级用户签到场景 |
| 地理位置 | GEO（基于 ZSet） | 附近的人 |

## 生产实践

### 常用场景

- **缓存**：最广泛的应用，降低 DB 压力
- **分布式锁**：`SETNX` + Lua 脚本或 Redlock
- **计数器**：`INCR` 做访问统计、限流
- **消息队列**：Stream (v5.0+) 替代 List 方案
- **Session 共享**：替代 Tomcat Session，多实例无状态

### 踩坑记录

1. **没有设置过期时间**：数据只增不减，内存撑爆。**每个写入都要问自己：这个 key 需要存活多久？**
2. **Big Key（大 Key）**：单个 Hash/List 超过万级元素，`HGETALL` / `LRANGE` 直接卡住 Redis 几秒。解决方法：拆分、换结构、用 `SCAN` 系列命令替代全量操作
3. **大量 key 同时过期**：Redis 过期清理是主动+惰性结合，同一秒内大量 key 过期会让 Redis 短暂 CPU 飙升。解决方案：过期时间加随机偏移 `TTL + random(0, 300)`
4. **`KEYS *` 是灾难**：生产环境严禁使用。用 `SCAN` 替代，游标式遍历不阻塞
5. **String 存 JSON 滥用**：对象直接用 String 存序列化 JSON 影响读写部分字段。用 Hash 可以只改某个字段

## 总结

五种基本结构各有擅长场景，选对结构比写对命令更重要。String + Hash 覆盖 80% 场景，ZSet + Stream 是高级玩具，用好了能简化大量业务逻辑。
