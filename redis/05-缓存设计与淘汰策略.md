# 缓存设计与淘汰策略

## 概述

Redis 最常见的使用场景是**缓存**——把数据库的热数据存一份到内存，挡住 90% 的读请求。但缓存不是"存了就完事"，设计不好反而拖垮系统。

## 设计原理

### 为什么用缓存

```
无缓存：
用户请求 → 查 MySQL → 返回
           MySQL 每秒只能抗几千 QPS
           100 个请求同时来 → MySQL 连接打满 → 卡死

有缓存：
用户请求 → 查 Redis → 命中 → 返回（微秒级）
                     → 没命中 → 查 MySQL → 写入 Redis → 返回
Redis 每秒抗 10 万 QPS，100 个请求毛毛雨
```

缓存的核心价值：**用空间（内存）换时间（响应速度），用廉价的 Redis 请求保护昂贵的数据库查询。**

### 缓存模式：旁路缓存（Cache-Aside）

最通用的模式：

```
读流程：
1. 查 Redis → 命中 → 直接返回
2. 没命中 → 查 DB → 写入 Redis → 返回

写流程：
1. 更新 DB
2. 删除 Redis 中的缓存（或更新）
```

**为什么写的时候是删缓存而不是更新缓存？**

- 更新缓存可能产生脏数据（并发写导致缓存和 DB 不一致）
- 删缓存成本低，下次读的时候再加载就是最新的

## 三大缓存问题

### 缓存穿透

**现象：** 请求的数据 Redis 和 DB 里都没有，每次请求都穿透到 DB。

```
黑客攻击：请求 GET /user?id=-1
Redis 没有 -1 → 查 MySQL 也没有 → 空结果不缓存
下次又来 GET /user?id=-2 → Redis 没有 → 查 MySQL...
大量不存在的数据请求直接把 DB 打挂
```

**解决：**

```bash
# 方案 1：缓存空值（设置短 TTL，比如 60 秒）
SET user:-1 "null" EX 60
# 下次同样的请求直接返回空，不打 DB

# 方案 2：布隆过滤器（Bloom Filter）
# 启动时把所有用户 ID 加载到布隆过滤器
# 请求来了先判断 ID 是否存在，不存在直接拒绝
```

```go
// Go: go-redis + 布隆过滤器（RedisBloom 模块）
exists, err := rdb.Do(ctx, "BF.EXISTS", "users", userID).Bool()
if !exists {
    return nil // ID 不存在，直接返回
}
```

### 缓存击穿

**现象：** 某个**热点 key** 刚好过期，大量并发请求同时穿透到 DB。

```
假设有个热门新闻的 key `news:hot`，TTL 5 分钟
过期的那一瞬间，同时来了 1 万个请求：
全部在 Redis 没查到 → 全部打到 MySQL → MySQL 挂了
```

**解决：**

```go
// 方案 1：互斥锁（只有一个请求去查 DB，其他等待）
// Go 实现缓存击穿防护
func GetHotNews(ctx context.Context, id string) (string, error) {
    val, err := rdb.Get(ctx, "news:hot:"+id).Result()
    if err == redis.Nil {
        // 没命中，抢锁
        lockKey := "lock:news:hot:" + id
        ok, err := rdb.SetNX(ctx, lockKey, "1", 5*time.Second).Result()
        if err != nil {
            return "", err
        }
        if ok {
            // 抢到锁，查 DB
            news := queryDB(id)
            rdb.Set(ctx, "news:hot:"+id, news, 5*time.Minute)
            rdb.Del(ctx, lockKey)
            return news, nil
        }
        // 没抢到锁，等待后重试
        time.Sleep(50 * time.Millisecond)
        return GetHotNews(ctx, id)
    }
    return val, nil
}

// 方案 2：热点 key 不过期（逻辑过期）
// 设置永不过期，后台定时更新
```

### 缓存雪崩

**现象：** 大量 key 在同一时间过期，或者 Redis 整个挂了，大量请求打到 DB。

```
场景 1：所有缓存 TTL 都设成了 1 小时
      第 1 小时整 → 全部过期 → 所有请求都打到 DB → DB 挂了

场景 2：Redis 集群宕机
      所有请求穿透到 DB → DB 跟着挂 → 系统瘫痪
```

**解决：**

```bash
# 方案 1：过期时间加随机偏移
SET user:1001 "..." EX 3600       # 错误，整点过期
SET user:1001 "..." EX 3600+RANDOM(0,300)  # 正确，分散过期时间

# 方案 2：Redis 高可用（哨兵/Cluster）
# 方案 3：降级——Redis 挂了直接返回旧缓存或默认值，不给 DB 压力
# 方案 4：限流——对 DB 查询做限流保护
```

## 内存淘汰策略

Redis 内存满了怎么办？由 `maxmemory-policy` 控制。

```conf
maxmemory 4gb
maxmemory-policy allkeys-lru
```

| 策略 | 行为 | 场景 |
|------|------|------|
| `noeviction` | 内存满了直接报错（默认） | 不允许丢数据的场景 |
| `allkeys-lru` | 淘汰最近最少使用的 key | 最常用，缓存通用 |
| `allkeys-lfu` | 淘汰最不常用的 key | 访问频率差异大的场景 |
| `volatile-lru` | 只淘汰设了 TTL 的 key 中最近最少用的 | 混合存储 |
| `volatile-ttl` | 淘汰 TTL 最短的 | 特殊需求 |
| `allkeys-random` | 随机淘汰 | 无所谓 |

生产推荐：**`allkeys-lru`**。够用、性能好、直觉上合理。LFU 复杂度高但效果更好，适合访问模式差异极大的场景。

## 生产实践

### 常用场景

- **页面缓存**：首页/商品详情，TTL 5~10 分钟
- **用户 Session**：TTL 30 分钟，每次访问续期
- **热点数据**：排行榜、配置参数，永不过期+定时刷新
- **限流计数器**：`INCR` + `EXPIRE`，比如 1 分钟内最多 100 次

### 布隆过滤器说明

布隆过滤器的特点是**只判断数据"绝对不存在"**——说过滤器说"没有"，那一定没有；说过滤器说"有"，可能误判（极小概率）。用这个特性来挡住穿透查询最有效。

### 踩坑记录

1. **缓存穿透被忽略**：以为"Redis 有缓存就够了"，没考虑数据不存在的情况。上线后被刷不存在的数据，MySQL 直接被打死。解决方案：缓存空值 + 参数校验
2. **TTL 设置不合理**：全部设置相同的 TTL 导致缓存雪崩，或者 TTL 太短（30 秒查一次 MySQL）。按业务特点定 TTL，加随机偏移
3. **大 Value 缓存**：把整个商品详情（包括图片 base64）序列化后塞 Redis，一个 value 几 MB，写入读取都慢。缓存只存需要频繁访问的字段，大对象存对象存储
4. **缓存和 DB 数据不一致**：写操作先更新 DB 再删缓存——删缓存失败了，下次读的还是旧数据。解决方案：延迟双删（先删缓存 → 更新 DB → 再延迟几百毫秒删一次），或用 Canal 监听 binlog 同步

## 总结

缓存设计的三板斧：**穿透要挡、击穿要锁、雪崩要分散**。再加一个 `allkeys-lru` 兜底防内存爆炸。记住：缓存不解决所有性能问题，它只是在 DB 前面加了一层保护。
