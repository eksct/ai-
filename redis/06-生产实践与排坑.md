# Redis 生产实践与排坑

## 概述

前面笔记讲了 Redis 的各个功能模块，这篇汇总生产环境下最常见的**真实踩坑案例**和**最佳实践**。每一条都是线上真实事故的总结。

## 大 Key（Big Key）问题

### 现象

单个 key 包含的数据过大——String > 10MB 或集合类型 > 万级元素。表现为 Redis 读写卡顿、请求超时。

### 排查

```bash
# 扫描大 Key（v4.0+）
redis-cli --bigkeys

# 看单个 key 占多少内存
MEMORY USAGE user:set:online

# 定位大 Key 类型
TYPE user:set:online
STRLEN user:set:online       # String 长度
HLEN user:set:online         # Hash 字段数
LLEN user:set:online         # List 长度
SCARD user:set:online        # Set 基数
ZCARD user:set:online        # ZSet 基数
```

### 危害

- 读大 Key：`HGETALL` / `LRANGE` / `SMEMBERS` 阻塞 Redis 几秒
- 写大 Key：写入时也需要同样时间
- 删大 Key：`DEL` 是同步的，删大 Key 也阻塞
- RDB 快照和加载都变慢

### 解决

```bash
# 方案 1：拆分（推荐）
# 把一个大 Set 拆散
大 Set: user:online → 500 万元素
拆成: user:online:1, user:online:2, ..., user:online:100
每个 5 万元素，读写都快

# 方案 2：换结构
# 需要全量获取 -> 用 SCAN 游标式遍历
SCAN 0 MATCH user:online:* COUNT 1000

# 方案 3：异步删除（v4.0+）
UNLINK user:bigkey   # 后台线程删，不阻塞主线程
```

## 热 Key（Hot Key）问题

### 现象

某个 key 被超高频率访问（比如 10 万 QPS 打在一个 key 上），导致单台 Redis 节点 CPU 打满。

常见场景：明星开播、秒杀商品、热搜榜单。

### 排查

```bash
# 开启热点 key 检测（v6.0+）
CONFIG SET hotkey-log-accurate yes

# 用 monitor 观察（慎用，高负载下别开）
redis-cli --hotkeys

# 或客户端统计
# go-redis 自带统计：rdb.PoolStats()
```

### 解决

```bash
# 方案 1：本地缓存 + Redis 多级缓存
# 在应用内存里（如 Go map 或 Java Caffeine）缓存热 key
# 先查本地 → 没命中再查 Redis → 再查 DB
# 一个热 key 只在每个应用实例上被查一次 Redis

# 方案 2：热 key 备份
# 把 key 复制多份，分散访问压力
news:hot → news:hot:1, news:hot:2, news:hot:3
# 客户端随机读一个副本
副本名 := fmt.Sprintf("news:hot:%d", rand.Intn(3))

# 方案 3：读写分离
# 把热 key 的读请求分散到从库
```

## 慢查询

### 排查

```bash
# 查看慢查询日志
SLOWLOG GET 10                 # 最近 10 条慢查询
SLOWLOG LEN                    # 慢查询总数

# 配置慢查询阈值
CONFIG SET slowlog-log-slower-than 10000  # 超过 10ms 的记录
CONFIG SET slowlog-max-len 1000          # 保留最近 1000 条
```

### 常见慢查询原因

| 命令 | 问题 | 替代方案 |
|------|------|---------|
| `KEYS *` | 遍历全部 key，阻塞 | `SCAN` 游标式遍历 |
| `HGETALL` 大 Hash | 返回所有字段 | `HSCAN` 分批，或拆小 |
| `LRANGE list 0 -1` | 返回全部列表元素 | 分页取 `LRANGE 0 99` |
| `SORT` | 复杂度 O(n+m*log(m)) | 业务层排序或用 ZSet |
| `ZRANGE` 大 ZSet | 一次取太多 | 分页查询 |

## 连接数管理

### 排查

```bash
# 查看当前连接
CLIENT LIST
INFO clients          # 连接数统计

# 最大连接数
CONFIG GET maxclients
```

### 解决

```bash
# 配置合理
maxclients 10000          # 根据内存和 CPU 调整
timeout 300               # 空闲 300 秒断连

# 客户端要有连接池
# Go: go-redis 默认池大小 10，可根据 QPS 调整
rdb := redis.NewClient(&redis.Options{
    PoolSize:     50,      # 连接池大小
    MinIdleConns: 10,      # 最小空闲连接
    MaxConnAge:   30 * time.Minute, # 连接最大存活时间
})
```

## 监控与告警

### 关键指标

```bash
# 内存
INFO memory
used_memory_peak            # 历史峰值，看是否需要扩容
used_memory                  # 当前使用
mem_fragmentation_ratio     # 碎片率 > 1.5 说明碎片严重

# 命中率
INFO stats
keyspace_hits / (keyspace_hits + keyspace_misses) # 命中率 > 90% 正常

# 延迟
redis-cli --latency -h 127.0.0.1 -p 6379  # 实时延迟监控
redis-cli --intrinsic-latency 100          # 本机延迟基准

# 持久化
INFO persistence
rdb_last_bgsave_status       # 上次 RDB 是否成功
aof_last_bgrewrite_status    # 上次 AOF 重写是否成功
```

### 告警阈值

| 指标 | 警告 | 严重 |
|------|------|------|
| 内存使用率 | > 80% | > 90% |
| 命中率 | < 90% | < 80% |
| 延迟 | > 10ms | > 50ms |
| 连接数 | > 80% maxclients | > 95% |
| 碎片率 | > 1.5 | > 2.0 |
| 慢查询 | > 10 条/分钟 | > 100 条/分钟 |

## 生产实践清单

### 必做事项

1. **设置 maxmemory**：防止内存写爆导致 OOM
2. **合理 TTL**：每个 key 都要问"什么时候过期"
3. **连接池**：客户端必须用连接池，禁止每次请求新建连接
4. **监控覆盖**：内存、命中率、延迟、慢查询、连接数
5. **定期备份 RDB**：即使开了 AOF，也要每天备份 RDB 到远程

### 坚决避免

1. **`KEYS *`**：生产用了直接事故
2. **大 Key 不过问**：用 `--bigkeys` 定期扫描
3. **不设 TTL**：内存只增不减，迟早爆炸
4. **所有请求走主库**：没有利用从库分担读压力
5. **Redis 裸奔**：至少设置密码 `requirepass`，绑定内网 IP

### 架构决策速查

| 数据量 | QPS | 架构 | 说明 |
|--------|-----|------|------|
| < 10GB | < 5 万 | 单机 | 纯缓存，关持久化省性能 |
| < 50GB | < 10 万 | 主从 + 哨兵 | 读写分离，高可用 |
| > 50GB | > 10 万 | Cluster | 数据分片，水平扩展 |

## 总结

Redis 生产环境的坑，80% 可以归结为三件事：**大 Key、热 Key、不设 TTL**。监控到位、定期扫描、配置合理，基本能避免绝大多数线上事故。
